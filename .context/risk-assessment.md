## HIGH Risks

### RISK-001: No E2E or API Test Coverage

| Field | Value |
|-------|-------|
| Severity | HIGH |
| Description | The target repo (`upex-bunkai-tms`) has 19 unit/integration tests in `lib/` covering validation logic, but zero E2E tests (no Playwright/Cypress config) and zero API integration tests from a black-box consumer perspective. The QA boilerplate (`raul-bunkai-qa-engineering`) has zero tests authored yet. |
| Impact | No automated regression gate. Any breaking change in auth flow, ATC creation, Jira import, or RBAC can reach production undetected. |
| Recommendation | This is the primary scope of the QA project. Prioritize: (1) auth flow E2E, (2) ATC CRUD API tests with each role, (3) Jira import job lifecycle tests. |
| Owner | QA team |
| Location | Filesystem scan — no `tests/`, `e2e/`, `playwright.config.*` found in target repo root |

### RISK-002: No CI Pipeline

| Field | Value |
|-------|-------|
| Severity | HIGH |
| Description | No `.github/workflows/` directory exists in the target repo. Tests (the 19 existing unit tests) only run when manually invoked (`bun test`). No automated test gate before deploy to staging or production. |
| Impact | Regressions can deploy to production without any automated check. The `repo:check` script exists but is not wired into any CI job. |
| Recommendation | Create a GitHub Actions workflow that runs `bun test` + `bun run repo:check` on every PR to `staging` and `main`. This is a QA team deliverable to propose to the dev team. |
| Owner | Dev + QA teams |
| Location | Filesystem scan — no `.github/workflows/` directory found |

### RISK-003: Single Supabase Project Shared Across All Environments

| Field | Value |
|-------|-------|
| Severity | HIGH |
| Description | All three environments (local, staging, production) share the same Supabase project (`fmbpikzpkafptqximhxn`). Test data created during QA sessions on staging writes to the same database that production reads from. |
| Impact | (1) Test-created workspaces, projects, ATCs can appear in production reads if multi-tenant isolation has any gap. (2) Test data accumulates in production DB. (3) QA tests that clean up test data risk deleting real data if cleanup logic is too broad. |
| Recommendation | (1) Always use workspace-scoped test data with a clearly QA-labeled workspace name (e.g., `qa-test-<timestamp>`). (2) Implement a test data cleanup routine that deletes only QA-labeled workspaces. (3) Request dev team create a separate Supabase project for staging — track as a QA finding. |
| Owner | Dev team (infra), QA team (test isolation) |
| Location | `target/.agents/project.yaml` lines 56, 60, 64 — same `db_project_ref` in all environments |

## MEDIUM Risks

### RISK-004: No Test Runner Script Defined in package.json

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| Description | The target repo's `package.json` has no `test` script. Running `bun test` likely works via Bun's auto-discovery, but it is not validated. The QA boilerplate needs to document the correct command. |
| Impact | QA automation cannot be invoked via a standard `bun run test` command without first verifying Bun auto-discovery works. |
| Recommendation | Verify `bun test` works in the target repo. Add a `test` script to `package.json` of the QA boilerplate. |
| Owner | QA team |
| Location | `target/package.json` — no `test` key in `scripts` section |

### RISK-005: Auth Test Credentials Not Documented

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| Description | No staging test user credentials exist in `.env.example` or any documentation. E2E tests require pre-provisioned test accounts. The `dbhub.toml` comment references "credentials Epic (BK-29)" but this was not accessible. |
| Impact | E2E test authoring is blocked until test credentials are available. |
| Recommendation | Request staging test credentials from the dev team. Add them to the QA boilerplate's `.env` as `LOCAL_USER_EMAIL`, `STAGING_USER_EMAIL`, `STAGING_USER_PASSWORD` etc. per CLAUDE.md Rule #1. |
| Owner | Dev team (provision), QA team (document) |
| Location | `target/dbhub.toml` comment + `target/.env.example` (no test credential keys) |

### RISK-006: Generated TypeScript Types Not Committed

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| Description | `lib/supabase-types.ts` (generated by `bun run types:gen`) is not committed to the target repo. TypeScript types for all Supabase tables are therefore unavailable without running the generator. |
| Impact | API test fixture types cannot be strongly typed without first running `types:gen`. |
| Recommendation | Run `bun run types:gen` in the target repo once environment is configured. Consider asking dev team to commit the generated file or provide it as an artifact. |
| Owner | Dev team |
| Location | `target/package.json` script `types:gen` — output file not found in repo snapshot |
