# SRS/ - Software Requirements Specification (Phase 2-3: Architecture & Infrastructure)

This directory contains **technical specifications** discovered from an existing project.

## Purpose

The `SRS/` folder stores technical implementation details:

- System architecture (components, connections)
- API contracts (endpoints, payloads)
- Functional specifications (what the system does)
- Non-functional requirements (performance, security)
- Infrastructure setup (deployment, CI/CD)

## Expected Files

| File                      | Description                            | Generated By                  |
| ------------------------- | -------------------------------------- | ----------------------------- |
| `architecture.md`         | System architecture diagrams (C4, ERD) | `/project-discovery` skill    |
| `functional-specs.md`     | Feature behavior specifications        | `/project-discovery` skill    |
| `non-functional-specs.md` | Performance, security, scalability     | `/project-discovery` skill    |
| `infrastructure.md`       | Deployment, CI/CD, environments        | `/project-discovery` skill    |

> **API contracts are NOT a file in this folder.** The technical surface lives in `api/openapi-types.ts` (generated by `bun run api:sync` from the project's OpenAPI spec). The business angle lives in `.context/business/business-api-map.md` (produced by the `/business-api-map` command). Do not hand-write a parallel `api-contracts.md`.

## How to Generate

Load the `/project-discovery` skill:
1. Run Phase 2 (Architecture) -- generates SRS architecture/API/functional/non-functional specs
2. Run Phase 3 (Infrastructure) -- complements `infrastructure.md`

### Prerequisites

- `.context/business/business-model.md` and `domain-glossary.md` must be populated (Phase 1)
- `.context/PRD/` must be populated (Phase 2 PRD prompts)
- Access to source code (backend, frontend)
- Access to database schema
- (Optional) Access to OpenAPI/Swagger specs
- (Optional) Access to CI/CD configuration

## Discovery Sources

Information in this folder is DISCOVERED from:

1. **Backend Code Analysis**
   - Route definitions → API contracts
   - Database models → data structure
   - Middleware → authentication/authorization

2. **OpenAPI/Swagger Specs**
   - Existing API documentation
   - Request/response schemas

3. **Database Inspection**
   - Schema analysis via MCP (DBHub)
   - Table relationships and constraints

4. **Infrastructure Files**
   - `.github/workflows/` → CI/CD pipelines
   - `docker-compose.yml` → service architecture
   - `Dockerfile` → build process
   - Cloud configs (terraform, serverless.yml)

5. **Configuration Files**
   - `package.json` → dependencies, scripts
   - Environment variable templates
   - Config files (next.config.js, etc.)

## When to Read This Folder

- **Phase 5 (Shift-Left Testing)**: API contracts for integration tests
- **Phase 12 (Test Automation)**: Technical details for automation setup
- **Phase 14 (Shift-Right Testing)**: Infrastructure for monitoring setup
- **Any API testing**: To understand endpoints and contracts
