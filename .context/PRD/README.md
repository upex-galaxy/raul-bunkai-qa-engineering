# PRD/ - Product Requirements Document (Phase 2: Architecture)

This directory contains **product-level requirements** discovered from an existing project.

## Purpose

The `PRD/` folder stores user-facing product information:

- What the product does (executive summary)
- Who uses it (user personas)
- How users interact with it (user journeys)
- What features exist (feature inventory)

## Expected Files

| File                   | Description                | Generated By                  |
| ---------------------- | -------------------------- | ----------------------------- |
| `executive-summary.md` | Product purpose and vision | `/project-discovery` skill    |
| `user-personas.md`     | Target user profiles       | `/project-discovery` skill    |
| `user-journeys.md`     | Key user flows and paths   | `/project-discovery` skill    |
| `feature-inventory.md` | Existing features catalog  | `/project-discovery` skill    |

## How to Generate

Load the `/project-discovery` skill and run its Phase 2 (Architecture) flow after completing Phase 1.

### Prerequisites

- `.context/business/business-model.md` and `domain-glossary.md` must be populated (Phase 1 completed)
- Access to source code (frontend routes, UI components)
- (Optional) Access to analytics data
- (Optional) Access to user research documents

## Discovery Sources

Information in this folder is DISCOVERED from:

1. **Frontend Code Analysis**
   - Route definitions reveal user journeys
   - Component structure reveals features
   - Form fields reveal user interactions

2. **Backend Code Analysis**
   - API endpoints reveal capabilities
   - Permission/role code reveals user types

3. **Existing Documentation**
   - Product specs in Confluence/Notion
   - User stories in Jira/Azure DevOps
   - Marketing materials

4. **Running Product**
   - UI exploration and screenshots
   - Feature testing

## When to Read This Folder

- **Phase 5 (Shift-Left Testing)**: Before creating test plans
- **Phase 10 (Exploratory Testing)**: To understand expected behavior
- **Any test documentation**: To map tests to features
