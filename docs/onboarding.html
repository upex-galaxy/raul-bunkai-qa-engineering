→Start here

Three onboarding paths, ordered by how much hand-holding the boilerplate gives you. The magic command is the recommended path — it downloads the latest template, scrubs upstream git history, renames the project, runs bun install, and hands off to the interactive installer in one shot.

Recommended · npm scaffolder

Magic command

bunx create-agentic-qa@latest <your-repo-name>

One command downloads the latest agentic-qa-boilerplate tarball (no git history), renames your project, runs bun install, and hands off to the interactive installer.

  1. Tarball download (latest main)
  2. Rewrite package.json + .agents/project.yaml
  3. Fresh git init -b main + initial commit
  4. bun install
  5. bun run setup — gentle-ai, 12 T1 skills, MCPs, .env, optional gh repo create

Details: npm package · source + flags

GitHub-first

"Use this template"

Click → clone → bun install && bun run setup

Create your repo on GitHub from day one. Good when you want the remote first and the local checkout second.

  1. Click Use this template → Create a new repository
  2. Pick owner, name, visibility
  3. Clone YOUR new repo locally
  4. bun install && bun run setup
  5. Manually edit package.json name + .agents/project.yaml project.project_name

Trade-off: keeps upstream git history; you rename manually. The magic command does both automatically.

Contributors only

Manual clone

git clone … && bun install && bun run setup

For hacking on the boilerplate itself — skills, installer, scripts, docs. End-users should NOT use this path.

  1. git clone https://github.com/upex-galaxy/agentic-qa-boilerplate.git
  2. cd agentic-qa-boilerplate
  3. bun install
  4. bun run setup

Use this only if you intend to send PRs against the boilerplate itself.

Why the magic command is the default

It is the only path that ends with a fully-renamed, gentle-ai-installed, MCP-wired, gh-repo-ready workspace in one command. The other two paths require manual rename + installer steps before the boilerplate is ready for the first /sprint-testing session.

Who you're talking to — AI personality

Before you fire your first command, know who is on the other side of the conversation. The AI in this repo has an opinionated, intentional personality — a deliberate mix of speech style, register, and behavioural defaults — so that what you read feels consistent across sessions, agents, and skills.

Short answer: a senior QA engineer who reports to a Project Manager. Competent, parsimonious with words, allergic to theatre, biased toward planning over improvisation, and disciplined about translating test work into business and quality value when speaking to you.

The six strategies you'll notice

Caveman mode

Word compression

Drops articles, fillers, and pleasantries. Fragments OK. Cuts roughly 75% of tokens without losing accuracy. Code blocks, commit messages, and security warnings stay fully written. Toggle with /caveman lite|full|ultra or disable with stop caveman.

Butler Pattern

Information granularity

Headline first (one line that resolves your literal question), then an atomic bullet menu of related topics. You pull the thread that matters. No paragraphs buried in walls of text — 12 specific bullets beats 3 broad buckets.

PM Voice Default

Vocabulary register

Headlines report user, business, or quality value — not technical action. Bullets are a SINGLE menu: each bullet's register (value-framed or technical) is chosen per topic, so a spec file path and an AC-impact statement can sit side by side. In foreground replies, a short attention-priming punch precedes the headline; when a menu has 3+ bullets, a short orientation question invites you to pull a thread. Auto-suspends when you mention a file, command, selector, or library by name, or when active skill is /sprint-testing · /test-documentation · /test-automation · /regression-testing · /framework-development.

Background-narrator

Lifecycle signaling

When running as a background job (scheduled regression runs, CI sweeps, queued sprint-testing), emits lifecycle signals (result:, needs input:, failed:) so a classifier can track progress without watching live. Never translated, never rephrased — they are a contract with the harness, not human comms.

Language mirror

Locale

Detects your language and replies in it. Repo artifacts (code, commits, PRs, branch names, Jira issues, ATC IDs, Xray entities) stay in English regardless of conversational language.

Visual Mapping Bias

Form

When the content is mappable — comparisons, hierarchies, flows, impact maps, test-execution states — the AI prefers a table, ASCII diagram, tree, box layout, or state machine over a paragraph of prose. The visual replaces prose, it does not decorate alongside it. Can sit below the headline or inside a single bullet. Plain ASCII preferred over Unicode box-drawing when terminal rendering is uncertain.

How to talk to it

  • Speak naturally in your own language. The AI mirrors you — no need to switch to English.
  • Drop a file path, command, selector, or library name in your message if you want a technical reply that turn (PM Voice auto-suspends).
  • State your goal (or the AC), not your implementation idea if you want the AI to push back when there's a simpler test path.
  • Ask for "technical mode" / "speak technically" (in any language) to force a technical reply for the rest of the session, or ask for "PM mode" / "PM voice" to restore the default.
  • Say "stop caveman" / "normal mode" (in any language) to disable token compression and read full-form prose.
Read the full guide before you start. The personality is layered: traits, strategies, override phrases, suspension triggers, and how to evolve it over time. Full reference (with all toggles, examples, and edge cases) lives in docs/ai-personality.md. It mirrors the runtime contract defined in CLAUDE.md sections 1–3.

1.The fundamental idea

This repository is an AI-driven QA machine. You bring a project under test (the target: a frontend, a backend, or both) and the boilerplate brings the instructions, the knowledge layer and the tooling that an AI agent uses to do quality engineering against it. The repo never runs alone — it is always paired with one specific target codebase.

For the machine to work, three families of moving parts cooperate: Skills orchestrate long, multi-phase AI work; Commands perform one-shot utilities; and Scripts are what you run yourself in the terminal to keep the configuration substrate healthy. All three feed off the same fuel — a system of variables defined in .agents/project.yaml that resolves to your target's specifics at AI session bootstrap.

The whole machine in one sentence

Three kinds of moving parts (Skills · Commands · Scripts) plus one variable substrate (.agents/project.yaml) — that is the entire mental model. The rest is detail.

Read agentic-quality-engineering.md for the strategy behind the practice and CONTEXT.md for why the knowledge layer is split the way it is. This page focuses on how the pieces connect — what to invoke, in what order, and what each piece actually does.

2.Three categories of moving parts

Everything you can invoke in this repo belongs to one of these three categories. The mental model holds end-to-end:

Type What it is Where it lives Who invokes it
Skill
Multi-file markdown that orchestrates the AI through long, multi-phase tasks (subagents, decisions, references). .claude/skills/<name>/SKILL.md + references/ The AI auto-activates when your prompt matches the skill's triggers. Force-load via /skill-name.
Command
Single-file markdown for one-shot, focused utilities. .claude/commands/<name>.md You explicitly type /<name>. No auto-trigger.
Script
TypeScript CLI that runs in the terminal. scripts/*.ts and cli/*.ts You run bun run <script> directly.

The three categories share the same fuel: the variable substrate in .agents/project.yaml (and Jira metadata in .agents/jira-fields.json + .agents/jira-workflows.json + .agents/jira-required.yaml). Skills resolve {{VAR}} and {{jira.*}} placeholders against it on every AI session; commands use the same resolver; scripts read the YAML and write back to it. See §5 Circuit 1 for the full resolution flow.

3.The lifecycle: five phases

Each phase has a clear trigger, a deterministic output and a frequency. You move forward when the previous phase's output exists; you loop back to Maintenance whenever documentation drifts from reality.

Read the table left-to-right: it tells you which phases are one-time (Bootstrap, Discovery, Adapt), which are continuous (Daily QA), and which kick in on drift (Maintenance) — so you know up-front what to expect on a fresh clone vs. an ongoing project.

0 · Bootstrap One-time per clone 15–30 min Fresh clone, env reset, or bun run setup re-run
1 · Discovery One-time per project 2–4 hours New target project, major product pivot, or /project-discovery re-run
2 · Adapt One-time post-Discovery 1–2 hours Stack change, KATA layer refactor, or /adapt-framework re-run

4.The three confusing pieces

Three pieces have similar-sounding names and overlapping verbs ("init", "adapt", "refresh"). They are not interchangeable. Use this table to avoid running the wrong one.

/adapt-framework
(skill + command) 		Once per target, AFTER /project-discovery finishes. Adapts tests/components/, config/, api/schemas/ to the target's stack: fixtures, page objects, API clients. Modifies THIS repo only — never the target. Plan → user approval → Implement. Once per target
/sync-ai-memory
(command) 		Periodically, when .context/ drifts and AI-critical docs no longer reflect the project state. A passive role you never invoke

agentic-qa-core has a second life beyond init: it hosts shared references that workflow skills cite on demand. When /sprint-testing, /test-documentation, /test-automation, or /regression-testing delegates to a subagent, the AI loads agentic-qa-core/references/briefing-template.md, dispatch-patterns.md, and orchestration-doctrine.md automatically. You never type /agentic-qa-core to read those files — they are pulled in as part of orchestration. This passive role is invisible from the user's seat, but it is why agentic-qa-core exists even after Phase 0 is complete.

5.How everything connects internally

Three circuits explain the wiring. Circuit 1 is about variable resolution (the static substrate). Circuit 2 is about subagent orchestration (the dynamic execution model). Circuit 3 is about CLI auto-load — which skill must be in context before a shell binary runs.

Circuit 1 — Variables

Variables flow: you fill the YAML once → AI reads it on every session → placeholders resolve to real values before any tool call.

.env vs .agents/project.yaml — two separate systems

Do not conflate them. They have different consumers, different lifecycles, and only one of them is committed.

.env .agents/project.yaml
Purpose Playwright + KATA runtime secrets and config. AI context-engineering variables for {{VAR}} resolution.
Consumers Test runner (bun run test, config/variables.ts, fixtures, login helpers, tests/utils/jiraSync.ts). AI agents (Claude Code, Cursor, Codex, Copilot, OpenCode) when resolving skill / template / doc references.
Examples LOCAL_USER_EMAIL, STAGING_USER_PASSWORD, XRAY_CLIENT_SECRET, HEADLESS, DEFAULT_TIMEOUT. PROJECT_KEY, WEB_URL, API_URL, JIRA_URL, DB_MCP, default_env.
Secrets? Yes (passwords, tokens, API keys). No — must remain commit-safe.
Committed? Gitignored. .env.example is the committed template. Committed.
Lifecycle Edited per developer / per CI runner. Edited once when adopting the boilerplate; rarely changes after.

The variable substrate has several syntaxes that look similar but resolve from different files. The full contract lives in .agents/README.md; the short version:

  • {{VAR_NAME}} → resolves from .agents/project.yaml (project-level static config).
  • <<VAR_NAME>> → session variable, computed at runtime by the calling skill (e.g. <<ISSUE_KEY>> extracted from the git branch). Never persisted.
  • {{jira.<slug>}} → portable Jira custom-field reference, resolves through .agents/jira-required.yaml (manifest) + .agents/jira-fields.json (workspace catalog). Sub-forms: {{jira.<slug>.<option>}} (option value for plain option / array-of-option fields), {{jira.<slug>.<parent>.<child>}} (cascading option-with-child).
  • {{jira.work_type.<slug>}} / {{jira.status.<work_type>.<slug>}} / {{jira.transition.<work_type>.<slug>}} → portable Jira workflow references. work_type resolves to the literal issue-type name (e.g. "Story"); status resolves to the literal status name (sub-keys: .id, .category); transition resolves to the transition id (sub-key: .name) — use the id form to invoke transitions unambiguously via REST. All resolve through .agents/jira-required.yaml work_types: (manifest) + .agents/jira-workflows.json (workspace catalog).

Circuit 2 — Orchestration

Each workflow skill declares its own dispatch points; the canonical briefing template + dispatch patterns live in agentic-qa-core/references/ so subagents load them once, fresh.

This is the canonical pattern the four workflow skills follow. The orchestrator reads its own dispatch strategy, writes a 7-component briefing for each subagent (Goal · Context docs · Project Standards (auto-resolved) · Skills to load · Exact instructions · Report format · Rules), and the subagent loads tool skills (/acli, /xray-cli, /playwright-cli) to actually run shell commands. The Project Standards block is pre-extracted by the orchestrator from .claude/skills/REGISTRY.md (built by bun run skills:registry) so subagents trust the compact rules and skip re-reading every full SKILL.md. This is what keeps the main conversation lean — the heavy reading happens inside subagents.

Readiness preflight gate (t=0)

Before any real work, the six testing skills (/shift-left-testing, /sprint-testing, /test-documentation, /test-automation, /regression-testing, /framework-development) run a mandatory readiness probe — missing tools, dead MCPs, expired tokens, unset credentials, or an unadapted framework are caught at the start instead of mid-run. Each skill declares only its own capability delta; the shared baseline lives in agentic-qa-core/references/preflight-gate.md.

Circuit 3 — CLI auto-load

Every shell binary the AI may run has an owning skill that must be in context first — the skill carries the binary's flag catalog, auth flow, pagination quirks, and error semantics. The orchestrator checks the mapping before each Bash call and loads the missing skill if needed. Source-of-truth: CLAUDE.md §6.5 + cli/install.ts.

CLI binary Skill(s) loaded BEFORE the call
gh /gh-cli (community USER) + /git-flow-master (in-repo, when the command is git / PR-shaped)
acli /acli (in-repo)
playwright-cli /playwright-cli + /playwright-best-practices (community PROJECT)
bunx allure (run / agent / generate / open / watch) /regression-testing + /test-automation (in-repo)
bun xray /xray-cli (in-repo)
jq /acli (primary consumer of jq pipelines)
resend /resend-cli (community USER, when added)
bun runtime — no skill mapping

Why this circuit matters

A raw CLI invocation that skips the owning skill loses the project-specific patterns the skill encodes (auth flow, error handling, pagination, gotchas). The mapping is enforced even when the user types the binary verbatim — the orchestrator loads the matching skill first, then runs the command.

Engram — cross-session memory the AI manages for you

What Engram is

Engram is an MCP server that persists structured memory across sessions and compactions. The AI calls mem_save proactively (no user prompt needed) whenever it makes an architecture decision, fixes a bug, learns a project convention, or hits a non-obvious gotcha. Subsequent sessions recall those memories via mem_search + mem_get_observation, so "we decided to vet the API for pagination drift before automating it" survives without you having to re-explain it.

What the AI saves automatically: architecture / design decisions, conventions, bug root causes, non-obvious discoveries (e.g. flaky-test patterns, KATA-fixture choices), workflow changes, user preferences, project-specific gotchas.

What the AI searches automatically: any message containing "remember", "recall", "what did we do", or any reference to past work — in any language you write in.

Session close: before saying "done", the AI calls mem_session_summary with Goal · Discoveries · Accomplished · Next Steps · Relevant Files. That summary is the seed for the next session. Skip it and the next session starts blind.

Full protocol: global CLAUDE.md § Engram. Installed by gentle-ai at user level (T2) via cli/install.ts — not committed in this repo.

→Choose your path

Four gates determine which workflow skill you invoke on a given ticket or release. They are NOT mutually exclusive in a project's lifetime — most projects flow 0 → A → B → C as a feature matures from a raw backlog idea to automated regression. They ARE mutually exclusive within a single session: you pick one entry point and let the skill drive.

Path 0 · Pre-sprint

/shift-left-testing (Stage 0)

Batch grooming of backlog Stories BEFORE sprint planning. Produces refined ACs + ATP DRAFT + PO/Dev question set per Story, transitions backlog → shift_left_qa → estimation.

  • Stories sit in Backlog / Shift-Left QA / Estimation / Ready For Dev
  • ACs are sparse or ambiguous — PO needs help refining
  • Feature does not exist yet — NO execution, NO smoke, NO DB queries
  • Output: label shift-left-reviewed + batch report to PO + Dev lead

Tool skills loaded: /acli (+ /xray-cli if Modality A). Downstream: /sprint-testing short-circuits Phases 1-3 when the label is fresh (<30 days).

Path A · Manual

/sprint-testing (Stages 1–3)

Per-ticket exploratory + verification work. Produces ATP, ATR, and bug reports in the TMS.

  • Jira ticket is "Ready For QA" or a bug retest
  • ATP + ATR artifacts must land in the TMS
  • Risk-triage decision tree applies on every bug
  • UI / API / DB triforce exploration is relevant

Tool skills loaded: /acli, /xray-cli, /playwright-cli.

Path B · Codify

/test-documentation → /test-automation (Stages 4–5)

Tests worth keeping. ROI > threshold. The test case is promoted from manual to automated and added to the regression suite.

  • Test case passed ROI gate (Candidate verdict)
  • AC is stable — no churn expected near-term
  • Automation can reach the surface (UI / API / DB)
  • Adds value to the regression suite, not noise

Tool skills loaded: /xray-cli, /playwright-cli, /playwright-best-practices.

Path C · Regression

/regression-testing (Stage 6)

CI suite execution. Pre-release or nightly run. Produces a stakeholder report with GO / CAUTION / NO-GO verdict.

  • Pre-release or nightly trigger
  • Existing automated regression suite is in place
  • Need failure classification (REGRESSION / FLAKY / KNOWN / ENVIRONMENT / NEW)
  • Stakeholder release-readiness report needed

Tool skills loaded: bunx allure, gh.

Framework evolution is NOT a QA path

Modifying KATA bases, fixtures, cli/, scripts/, or api/schemas/ goes through /framework-development — a self-contained Plan → Code → Verify → Archive orchestrator that needs no SDD-* skills (those are opt-in only if a contributor installs them manually). NEVER run framework-evolution work through /sprint-testing, /test-documentation, /test-automation, or /regression-testing. The boundary is enforced inline in each workflow skill.

7.Skills catalog

Every T1 skill that ships committed in .claude/skills/ — the skills we maintain in this repo. Some you invoke directly; others (the tool skills and the foundation) load automatically when a workflow skill needs them. Community skills (T3/T4 — playwright-cli, playwright-best-practices, resend-cli, skill-creator, wokitoki, etc.) are not listed here; the installer fetches them from upstream at setup time. Legend — Category vs Tier

Every skill carries two badges. Category says what it does (the role it plays in the pipeline). Tier says where it lives and how it was installed.

Category — what it does

Foundation hosts shared references and bootstraps the repo.

Workflow drives a stage of the pipeline.

Generator reverse-engineers context from the target.

Tool wraps a CLI / MCP for I/O.

Reference is loaded passively by other skills.

Tier — where it lives

T1 Project-owned, committed in .claude/skills/ (the 12 listed below).

T2 gentle-ai dependency at user level via cli/install.ts (Engram, SDD-*).

T3 Community project-level installed by cli/install.ts (playwright-cli, playwright-best-practices).

T4 Community user-level — ALWAYS ASK before loading.

agentic-qa-core Foundation T1

Skill composition — how skills load each other

Read top to bottom. /framework-development is the only self-contained orchestrator for boilerplate evolution; per-ticket workflow skills do not compose each other — they hand off sequentially via the 6-stage pipeline. Tool skills sit at the bottom of the food chain.

8.Commands & Scripts

One-shot utilities — neither orchestrate long flows like skills nor maintain config like setup scripts. You type them by name.

Commands

Single-file markdown at .claude/commands/<name>.md. Invoke with /<name>.

  • /adapt-framework Adapt KATA architecture (tests/, api/schemas/, config/, CI, MCP) to a discovered target. 10-phase idempotent flow (Phase 0 prereq/genericness gate → Phase 9 scan): no writes before approval; re-runs report a GENERIC / ADAPTED checklist.
  • /sync-ai-memory Sync all AI-critical docs (CLAUDE.md, README.md, INSTALLER.md, CONTEXT.md, docs/*.md, and this onboarding page) against current .context/ + package.json.
  • /business-data-map Generate or refresh .context/business/business-data-map.md — entities, flows, state machines.
  • /business-feature-map Generate or refresh .context/business/business-feature-map.md — feature catalog, CRUD matrix, integrations.
  • /business-api-map Generate or refresh .context/business/business-api-map.md — auth model, critical endpoints, architecture.
  • /master-test-plan Generate or refresh .context/master-test-plan.md — what to test and why.
  • /break-down-tests Plain-English breakdown of automated tests for a given module / spec.
  • /fix-traceability Repair broken US ↔ ATP ↔ ATR ↔ TC traceability links in the TMS.

Scripts

TypeScript CLIs declared in package.json. Run with bun run <name>.

  • bun run env Canonical set -a; source .env; set +a snippet. Run it inline in your terminal (not via bun run, which uses a child subshell) to export every .env key into the current shell — then bare claude, opencode, acli, bun xray all inherit the vars with no wrapper.
  • bun run agents:setup Interactive walkthrough that fills .agents/project.yaml.
  • bun run setup --variables Idempotent env-variable setup menu — set / reset the critical tool credentials (Atlassian, Resend, Tavily) in .env, walk every variable one-by-one, or push the local .env to GitHub Actions secrets via gh secret set. Scriptable with --variables-local / --variables-remote / --yes / --force. Remote writes are gated behind an explicit confirm and refused in CI; no secret value is ever printed.
  • bun run vars:env:check Validate .env against the typed variables manifest (cli/lib/variables-manifest.ts) — enforces manifest ↔ .env.example parity. Wired into repo:check.
  • bun run vars:check Validate every {{VAR}}, {{jira.<slug>}}, {{jira.work_type.*}}, {{jira.status.*}} and {{jira.transition.*}} reference resolves against the manifest + catalogs.
  • bun run jira:sync-fields Catalog all custom fields from the live Jira workspace into .agents/jira-fields.json (incl. option values for option and cascading option-with-child fields).
  • bun run jira:sync-workflows Catalog Jira workflow statuses + transitions per work_type (Story, Bug, Test) into .agents/jira-workflows.json. Interactive on first run; idempotent thereafter.
  • bun run jira:sync-link-types Catalog the Jira issue-link types into .agents/jira-link-types.json so traceability links (ATP / ATR / defect) resolve to the right relationship name.
  • bun run jira:sync-issues Registry-driven pull of Jira issues into .context/PBI/ markdown (default scope: Epics + Stories + Bugs). Each coverable type (Story / Bug / Defect / Improvement / Tech Story / Tech Debt) gets its own folder with auto-discovered ATP / ATR (a linked Xray Test Plan / Execution description overrides the custom-field copy), nested defects, and traceability + orphan-defect warnings. Sub-commands get <KEY> (one issue, all custom fields; --include-comments) and jql "<query>" (batch); flags --sprint, --types, --no-defects, --project (env defaults JIRA_SYNC_SPRINTS / JIRA_SYNC_TYPES; flag > env > default).
  • bun run jira:check Verify the Jira workspace satisfies .agents/jira-required.yaml — covers both the custom-field catalog and the work-type / workflow catalog.
  • bun run test:env:check Validate .env for the active TEST_ENV.
  • bun run api:sync Sync OpenAPI spec from the target API and regenerate api/schemas/ types.
  • bun run kata:manifest Extract ATCs from the codebase into a manifest. kata:manifest:watch for live watch mode.
  • bun run pw:install Install Chromium with system deps via Playwright.
  • bun run onboarding Serve this onboarding page locally and auto-open it in the browser.
  • bun xray Xray TMS sync: import / export / sync test results, executions, plans.
  • bun run test / test:e2e / test:integration / test:smoke Run the Playwright suite — full, E2E project, integration project, or @critical-tagged smoke.
  • bun run allure:run Generate and open the Allure report from ./allure-results.

Run bun <script> --help on any script for usage. The Allure binary is installed locally — no global install needed.

Artifact refresh cycle

Some commands and scripts produce living artifacts — files in .context/, api/schemas/, or kata-manifest.json that need to be re-run when their source of truth drifts. The mapping below tells you which trigger refreshes which artifact and when to fire it.

Trigger Artifact path Re-run when
/business-data-map .context/business/business-data-map.md Domain model evolves (new entities, state machines, flows, integrations)
/business-feature-map .context/business/business-feature-map.md Feature catalog drifts (new CRUD surfaces, integrations)
/business-api-map .context/business/business-api-map.md API surface or auth model changes
/master-test-plan .context/master-test-plan.md Scope or risk model shifts
/sync-ai-memory README.md, CLAUDE.md, INSTALLER.md, CONTEXT.md, docs/** After any structural change to skills, scripts, or docs
bun run api:sync api/schemas/ (TypeScript types from OpenAPI) API spec changes (technical sync, not business angle)
bun run kata:manifest kata-manifest.json New Component or ATC added; pre-commit hook blocks the commit when stale
/fix-traceability TMS US ↔ ATP ↔ ATR ↔ TC links Broken or missing traceability detected in the TMS

Rule of thumb — idempotent refresh

All triggers above are idempotent: they read current state, regenerate output, and overwrite the file. Safe to re-run anytime; they never accumulate drift. When in doubt — re-run.

9.TL;DR mental model

  • Skills = play scripts for the AI (multi-act, with characters that delegate).
  • Commands = single-act utilities you type by name.
  • Scripts (bun run …) = what YOU run in the terminal to keep the fuel (.agents/project.yaml + Jira fields) healthy.
  • .context/ = what the AI KNOWS about the target (generated by /project-discovery, kept fresh by /sync-ai-memory and the /business-*-map commands).
  • tests/ = the real code, adapted by /adapt-framework and authored via /test-automation.

Fill project.yaml first (bun run agents:setup), then let the AI discover the target with /project-discovery, then wire the tests with /adapt-framework. From there, invoke the four workflow skills depending on the QA stage you are in.
