why this matters

retries: 0

The flaky-test tax — KATA's local default. If a test passes only on retry, it is flaky: a bug to investigate, not a pass to accept. No arbitrary waitForTimeout, no shared state, every test makes its own data.

Lead with the cultural rule that defines the whole method. Flakiness is the silent killer of automated suites; KATA refuses to hide it behind retries.

The Craft
KATA by hand

If the AI did not exist, what would a QA engineer do, step by step? Plan before code. Respect the layers. One ATC = one test case. Review every line against the checklist.

Everything in Part 1 is the real method a human follows. The AI later automates this exact sequence — so master it first.

plan phase — read first, in order

Inputs You Gather Before Drafting Anything

  • kata-manifest.json (root). Authoritative registry of every Component and @atc('ID'). Load FIRST to avoid duplicates and ID collisions.
  • KATA references. Doctrine for layers, ATC identity, fixture selection, import aliases.
  • tests/components/. The existing Api/Page/Steps shapes on disk to match.
  • The synced Story folder. Read the ENTIRE folder — ACs, scope, business rules, AND comments — not one field.
  • api/schemas/ + .env. OpenAPI-derived types (refresh if stale); credentials via config.testUser — never hardcode.
The manifest is the anti-duplication gate (Critical Rule #12). Skipping it produces duplicate Pages, duplicate ATC IDs, and missed reuse — all rejected at review.

the heart of KATA — what is an ATC?

ATC = Complete Test Case, Not a Click

❌ NOT an ATC

  • clickLoginButton() — single interaction
  • fillEmailSuccessfully() — wraps one page.fill()
  • submitFormSuccessfully() — an interaction, not an outcome
→

✅ A real ATC

  • loginWithValidCredentials(data) — goto, fill, submit, assert redirect
  • createOrderSuccessfully(payload) — POST, assert 201, assert body.id, return tuple

If the method one-line-wraps page.click(), delete it.

The single most important reframe. An ATC is an atomic mini-flow with its own fixed assertions, decorated @atc('TICKET-ID'), mapping 1:1 to a TMS ticket.

TC identity rule

One TC = One Precondition + One Action

The definition

A test case is exactly two things: the PRECONDITION (state) and the ACTION (trigger). EVERY expected result from the same precondition + action is an assertion of the SAME TC — no matter which panel, endpoint, or section it validates.

❌ WRONG

  • TC-A: verify Pricing block
  • TC-B: verify Reviews block
  • TC-C: verify page structure
→

✅ RIGHT

  • One TC: "Open product detail page"
  • Pricing, Reviews, structure = its assertions
Splitting one action's assertions into multiple TCs is the most common over-counting mistake. It explodes the suite and breaks traceability. Collapse them.

equivalence partitioning

Same Output = One Parameterized ATC

❌ Three ATCs, one outcome

  • @atc('T1') loginWithWrongEmail()
  • @atc('T2') loginWithWrongPassword()
  • @atc('T3') loginWithEmptyFields()
  • all return HTTP 401
→

✅ One parameterized ATC

  • @atc('T1') loginWithInvalidCredentials(payload)
  • test file parameterizes the inputs
  • all produce 401 + error body

Actions inside change → different ATC. Only DATA changes but behavior is identical → same ATC, parameterized.

Different status codes, UI states, or business outcomes are separate ATCs. Same outcome, different data, is one. This keeps the ATC count honest.

test-design doctrine — the core principle

"All ACs covered" is the floor, not the success bar

1 · Passing ≠ probing

Green ACs confirm the contract. They don't prove the system is robust, safe, or error-resistant.

2 · Floor, not ceiling

"All ACs covered" is the entry bar. The ATC set must also cover risk-beyond-AC.

3 · Criterion vs case

A criterion is a business assertion. An ATC is its concrete exploration: precondition + action + assertions.

4 · 1 : N by default

One AC → many ATCs. EP-merge collapses WITHIN a partition only — never across partitions, boundaries, or states.

5 · Risk lives outside

Automate the invalid/boundary inputs, auth/error paths, state transitions, and anomalies the AC is silent on.

EP ≠ BVA

Same-behavior merge hides off-by-one. Wherever a range/limit/length/date-window exists, add explicit boundary ATCs.

Canon: agentic-qa-core/references/test-design-doctrine.md. The review gate (§3.4.1 input-domain, §3.4.2 state/temporal) enforces this in code.

EP without BVA misses off-by-one. A status field with no invalid-transition ATC is an incomplete plan. The triggers are binding.

ATC craft rules

Five Rules That Decide Acceptance at Review

  • ATCs are atomic. An ATC NEVER calls another ATC. Reusable chains live in a Steps module (no @atc).
  • Locators inline. No locators/*.ts. Reused in 2+ ATCs of the same component? Promote to a private readonly arrow function.
  • Helpers vs ATCs. Read-only GET is a helper (@step). A state-changing action is an ATC. A GET that verifies the action lives INSIDE the ATC.
  • Fixed inside, flow outside. Status / required fields / redirect go in the ATC; combined-outcome checks go in the test file.
  • Max 2 positional params. 3+ must use a named object param — fn(args: Args).
These five are the recurring rejection reasons. Internalize them and most of your code passes review on the first pass.

the architecture you must respect

KATA's Layers — Never Collapse Them

1

TestContext

config, logger, faker, environment accessors.

2

ApiBase / UiBase

HTTP helpers (typed tuple returns) and Playwright helpers.

3

Domain Components + Steps

UsersApi, LoginPage — @atc lives here. Steps = reusable ATC chains (no @atc).

4

Fixtures → Tests

Api/Ui/Steps/Test fixtures inject; tests orchestrate ATCs into scenarios.

A higher layer may use a lower one, never the reverse.

Flattening these layers is an automatic rejection. The separation is what lets API helpers, UI helpers, and agnostic utilities stay in their correct homes.

fixture selection — load-bearing every test

test data strategy — classify during PLANNING

Discover → Modify → Generate (Strict Priority)

1

Discover

Query the system for data already in the required state. Zero DB impact. Preferred.

2

Modify

Find existing data, mutate it via API into the required state. One mutation.

3

Generate

Create from scratch via faker payload + POST. Last resort; pollutes the env.

4

Blocker

No pattern feasible → flag NOT automatable, document the gap, escalate.

Override: use Generate even when data could be discovered IF the test mutates state (avoid order dependencies).

This classification happens in the plan, never deferred to coding. Discover runs in beforeAll; if it finds nothing the test uses test.skip — never an expect on precondition data.

the beforeAll trap

Never Put expect() Inside beforeAll

Watch out

A failing beforeAll assertion kills EVERY test in the describe block — including unrelated ones — and hides what really broke.

Right pattern

beforeAll DISCOVERS only (no assertions, populates describe-scope vars). Each test guards its own precondition: if (!order) return test.skip(true, 'No order available'). The report then distinguishes "skipped — missing data" from "failed — real bug". Cleanup: Discover needs none; Modify restores in afterAll; Generate deletes in afterEach.

A subtle but high-impact rule. Assertions in shared setup turn one missing fixture into a wall of red failures that obscures the actual signal.

code phase — the API component shape

A KATA API Component (Layer 3)

OrdersApi.ts
export class OrdersApi extends ApiBase {
  private readonly baseEndpoint = '/api/v1/orders';

  @step                                // helper — read-only, no @atc
  async getOrderById(id: string): Promise<[APIResponse, Order]> {
    return this.apiGET<Order>(`${this.baseEndpoint}/${id}`);
  }

  @atc('TICKET-ID')                      // ATC — state-changing
  async createOrderSuccessfully(payload: CreateOrderRequest) {
    const [response, body, sent] =
      await this.apiPOST(this.baseEndpoint, payload);
    expect(response.status()).toBe(201);  // FIXED assertion
    expect(body.id).toBeDefined();       // FIXED assertion
    return [response, body, sent];       // 3-tuple: enables chaining
  }
}
Note the order: baseEndpoint, constructor, helpers, then ATCs. GET/DELETE return 2-tuples; POST/PUT/PATCH return 3-tuples so tests can compare what was sent against what came back.

code phase — the UI component & locators

A KATA Page Component with the Locator Ladder

LoginPage.ts
export class LoginPage extends UiBase {
  // shared locator: used in 2+ ATCs → private readonly arrow (lazy)
  private readonly submitButton = () => this.page.getByTestId('login_submit_button');

  @atc('TICKET-ID')
  async loginWithValidCredentials(data: LoginData): Promise<void> {
    await this.page.getByTestId('email_input').fill(data.email);  // inline (used once)
    await this.page.getByTestId('password_input').fill(data.password);
    await this.submitButton().click();
    await expect(this.page).toHaveURL(/.*dashboard.*/);            // fixed assertion
  }
}
// Locator ladder:  1 getByTestId  2 getByRole  3 getByLabel  4 getByText  5 CSS/XPath
// BANNED: .btn-primary · div > form > button:nth-child(3) · page.waitForTimeout(3000)

Coming from the getByRole-first standard? Here data-testid is rung 1 because the dev team provides them as a stable testing contract — getByRole remains the immediate fallback.

Prefer data-testid every time it exists. CSS-class and structural selectors decay on every refactor. UI ATCs return void; assertions live inside.

tests orchestrate; they don't implement

The Test File — Flows, Not Property Checks

processCheckout.test.ts
import { test, expect } from '@TestFixture';   // NOT from @playwright/test

test.describe('UPEX-411: Apply Discount Code', () => {
  // RIGHT: ONE test = complete flow with multiple assertions
  test('UPEX-411: applies % discount when code is valid @critical',
    async ({ api }) => {
    const [, order] = await api.orders.createOrderSuccessfully(api.data.createOrder());
    const totals = await api.orders.getTotals({ orderId: order.id });
    expect(order.discountApplied).toBe(true);
    expect(totals.finalAmount).toBe(totals.baseAmount - totals.discountAmount);
  });
});
// WRONG: six tests each checking one field of the same response.
// Ticket-ID in describe AND every test(). One ticket per describe; never split across files.
Naming hierarchy: directory = module, file = {verb}{Feature}.test.ts, describe = ticket, test = scenario. Separate tests only for fundamentally different flows, preconditions, or roles.

review phase — the merge gate

A Clean Review Is the Merge Gate

  • Three shared checklists first. Component / ATC / Test-file review — naming, layer extension, decorators, tuple returns, inline locators, ticket-ID prefix.
  • Then the delta. E2E delta (locator quality, condition-based waits, session reuse) OR API delta (facade compliance, tuple discipline, status coverage, auth).
  • Coverage gates the handoff. Beyond HTTP status: §3.4.1 input-domain (EP + BVA boundaries) and §3.4.2 state-transition / idempotency / concurrency / timeout must be covered or marked explicitly N/A.
  • Severity model. CRITICAL & HIGH block merge; MEDIUM recommended; LOW nice-to-have. A hardcoded waitForTimeout or missing @atc is CRITICAL.
  • Revision ceiling. Max 2 revision loops. Still not APPROVED after 2 rounds? Escalate to the user; never loop forever.
  • Optional adversarial gate. For high-risk changes (new fixtures, shared base edits) run a dual-judge review before commit.
Every failed checklist item is a blocker. The handoff checklist ('ready for CI') must be all-true: zero CRITICAL, zero HIGH, all three gates green.

quality gates G1–G4 & the bug protocol

When a Test Uncovers a Real Bug

Named gates block progression

G1 Plan exists → G2 Tests pass → G3 Review APPROVED → G4 Progress updated.

If G2 fails on a REAL product bug

(1) search the tracker for an existing bug; (2) if reported, annotate test.fail('Blocked by {BUG-KEY}') or tag @blocked:{BUG-KEY} and keep it; (3) if not, show the observed-vs-expected diff, get confirmation, then file the bug; (4) record it in PROGRESS.md. Only THEN may G2 be soft-overridden. A failing test with no bug key behind it is never an acceptable override.

This keeps a found bug visible and traceable instead of silently skipped. The @blocked tag lets regression-testing exclude it from GO/NO-GO until the fix lands.

// takeaways

Three Rules That Keep a Suite Trustworthy

1 · One ATC = one test case

An atomic mini-flow with fixed assertions, decorated @atc('TICKET-ID'), never calling another ATC. Same output + different data = one parameterized ATC.

2 · Plan → Code → Review, always

Never jump to code; never skip the manifest check; never merge with a CRITICAL/HIGH finding or a non-green gate.

3 · Stability is discipline

retries: 0, no hardcoded waits, every test makes its own data, no expect() in beforeAll. A flaky pass is a bug.

If your audience remembers only three things, these are them. They are the difference between a suite people trust and one they ignore.

the payoff

@atc('TICKET-ID')

1:1 traceability, end to end. Every ATC links to a real TMS ticket; results flow back via NDJSON → atc_results.json → TMS sync. The manifest knows what exists in code; the result file knows what ran. Together: coverage you can defend to a PO.

This traceability is what elevates KATA from 'tests that pass' to 'evidence of what is covered and why' — the artifact a quality conversation actually needs.
