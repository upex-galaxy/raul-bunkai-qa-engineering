Concepto · 1 de 18

¿Qué es Xray?

Xray es una app de gestión de pruebas que vive dentro de Jira. No es una herramienta aparte: añade tipos de issue nuevos al mismo Jira que ya usas.

QA planifica, ejecuta y reporta sin salir del tablero — las pruebas conviven con las historias y los bugs.

La idea que lo cambia todo

Una prueba
es un issue.

Un caso, un plan, una ejecución… cada cosa es un issue Jira con su clave (CALC-123), su estado, su asignado y sus enlaces.

Clave: Xray no reinventa Jira, lo extiende. Todo lo que crea son issues normales → heredan permisos, workflows, JQL, dashboards. Fuente: docs.getxray.app — Xray Work Item Types.

Concepto · 2 de 18

«Todo es un issue» — y por qué importa

JQL

Buscas pruebas, planes y ejecuciones con el mismo lenguaje de búsqueda de Jira.

Permisos

Quién ve o edita una prueba lo decide el esquema de permisos del proyecto.

Enlaces

Una prueba se enlaza a una historia igual que un bug se enlaza a una historia.

Tableros

Las ejecuciones pueden aparecer en el board ágil junto a Stories y Bugs.

Dashboards

Los gadgets de Jira leen los datos de Xray sin integración extra.

Resultado

Cero contexto perdido. Requisito, prueba y defecto viven en el mismo lugar.

Ventaja estructural frente a TMS externos: al ser issues nativos, no hay sincronización ni doble fuente de verdad.

Las 6 piezas de Xray

Seis tipos de work item. Cada uno responde a una pregunta distinta.

Test

El QUÉ. La especificación de un caso de prueba.

Precondition

El ANTES. Un estado inicial reutilizable.

Test Set

El GRUPO. Una carpeta lógica para organizar pruebas.

Test Plan

El SEGUIMIENTO. El paraguas que consolida el progreso.

Test Execution

El CUÁNDO. La tarea de correr pruebas y recoger resultados.

Sub-Test Exec

El MISMO, pero hijo de un requisito, para el board.

Mnemotécnica: cada pieza responde una pregunta → QUÉ / ANTES / GRUPO / SEGUIMIENTO / CUÁNDO. Fuente: docs.getxray.app — purpose of each Xray entity.

Las piezas · 7 de 18

Test Execution  y el Test Run

Test Execution

Tarea asignable de correr un grupo de pruebas, para una versión y un entorno. Contiene varios Test Runs.

Test Run

Registro de UNA corrida de UNA prueba. Aquí viven estado, evidencia, defectos y el estado por paso.

Estados de una corrida

TODO EXECUTING PASS FAIL ABORTED
Execution = la sesión de corridas; Run = una línea dentro. Estados: TODO/EXECUTING/PASS/FAIL/ABORTED. Fuente: docs.getxray.app — Execute Tests.

El flujo · 9 de 18

El flujo natural de Xray

① ESPECIFICAR ② ORGANIZAR ③ PLANIFICAR Test (caso) → Test Set + → Test Plan Precondition (alcance) │ │ └─────────────────┐ ▼ ④ EJECUTAR ──────► Test Execution │ (Test Runs) ▼ ⑤ REPORTAR ──────► PASS / FAIL + 🐞 Defectos │ ▼ ⑥ CUBRIR ────────► Cobertura (OK / NOK)

Mnemotécnica · «ÓPERA»

Especificar · Organizar · Planificar · Ejecutar · Reportar · Cubrir. Cada prueba es una función que sube al escenario, actúa y deja crítica.

Orden lógico, no rígido: una ejecución puede nacer ad hoc o clonarse de un Plan previo. Fuente: docs.getxray.app — Process / Executing Tests.

El flujo · 10 de 18

Cómo se conectan las piezas

STORY CALC-10 "Pagar con tarjeta" │ la cubre / "is tested by" ▼ TEST CALC-201 ─┐ TEST CALC-202 ─┤── agrupados en ──► TEST SET "Pagos" TEST CALC-205 ─┘ │ incluidos en el alcance ▼ TEST PLAN CALC-300 "Sprint 14·Pagos" │ genera ciclos de corrida ▼ TEST EXECUTION CALC-400 ── contiene ──► Test Runs │ ▼ al fallar una corrida BUG CALC-455

Story se cubre con Tests → entran en un Plan → el Plan genera Executions → cada Run puede parir un Bug.

Columna vertebral. La Story es un issue Jira normal (un "requisito coverable"). El resto son issues Xray. Fuente: docs.getxray.app — Requirement Traceability Report.

Cobertura · 11 de 18

¿Está probado el requisito?

Xray calcula un campo «Requirement Status» en cada Story desde el estado de sus pruebas.

OK

Todas las pruebas pasaron.

NOK

Al menos una falló.

NOT RUN

Alguna sin correr (sin fallos).

UNCOVERED

Sin pruebas asociadas.

Mnemotécnica · «el semáforo + gris»

🟢 OK · 🔴 NOK · 🟡 NOT RUN · ⚪ UNCOVERED. La cobertura sube por jerarquía: si un hijo está NOK, el padre (Epic/Story) se vuelve NOK.

Cálculo: todas PASS→OK; alguna FAIL→NOK; alguna TODO sin fails→NOT RUN; sin pruebas→UNCOVERED. Hasta 3 niveles (Epic→Stories→Subtasks). Fuente: docs.getxray.app — calculation of coverage status.

Trazabilidad · 12 de 18

Trazabilidad: dos capas

Capa 1 · Enlaces Jira

El parentesco visible

Los issue links de Jira: "tests", "is tested by". Conectan Story ↔ prueba/plan/ejecución.

Capa 2 · Interna de Xray

El ADN de la corrida

La pertenencia propia de Xray: qué Tests hay en un Plan, qué Runs en una Execution, qué Defectos en un Run. No son issue links.

Mnemotécnica: «Jira enlaza ISSUES, Xray enlaza RESULTADOS». Por eso algo «se ve enlazado» en Jira pero la corrida no aparece en el Plan — son dos capas distintas.

Crítico para fix-traceability del repo: en jira-xray hay que reparar AMBAS capas. Capa 1 = acli/issue links; capa 2 = xray-cli (membership Test↔Plan↔Execution↔Run).

Defectos · 14 de 18

Los defectos en Xray

El bug nace en la corrida. Cuando una prueba falla, el tester lo crea desde la ejecución y queda enganchado a ese Test Run.

  • Es un issue Jira de tipo Bug normal.
  • Se enlaza al Test Run (no al Test abstracto).
  • Aflora en la columna Defectos del reporte.

JQL de defectos en Xray

// bugs al ejecutar un requisito issue in defectsCreatedForRequirement("CALC-10") // bugs durante una ejecución issue in defectsCreatedDuringTestExecution(CALC-400)

Xray sabe qué bug salió de qué corrida → traza bidireccional prueba ↔ defecto.

"El bug es hijo del Run, no del Test": un mismo Test puede tener 0 bugs en una corrida y 1 en otra. Fuente: docs.getxray.app — Execute Tests, defectsCreatedForRequirement / defectsCreatedDuringTestExecution.

Para llevarte · 17 de 18

Cinco ideas para no olvidar

1 · Todo es un issue. Xray vive dentro de Jira; hereda JQL, permisos y enlaces.

2 · ÓPERA. Especificar·Organizar·Planificar·Ejecutar·Reportar·Cubrir.

3 · Plan observa, Execution corre. El Run es la unidad real de resultado.

4 · Semáforo + gris. OK·NOK·NOT RUN·UNCOVERED = la cobertura.

5 · Dos capas de traza. Jira enlaza issues; Xray enlaza resultados. El bug es hijo del Run.

En este repo: ATP→Plan · TC→Test · ATR→Execution. Plan primero, siempre.
