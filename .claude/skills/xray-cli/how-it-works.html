Concept · 1 of 18

What is Xray?

Xray is a test-management app that lives inside Jira. It is not a separate tool: it adds new issue types to the same Jira you already use.

QA plans, executes, and reports without leaving the board — tests live alongside stories and bugs.

The idea that changes everything

A test
is an issue.

A case, a plan, an execution… each thing is a Jira issue with its key (CALC-123), its status, its assignee, and its links.

Key: Xray does not reinvent Jira, it extends it. Everything it creates are normal issues → they inherit permissions, workflows, JQL, dashboards. Source: docs.getxray.app — Xray Work Item Types.

Concept · 2 of 18

“Everything is an issue” — and why it matters

JQL

You search tests, plans, and executions with the same Jira search language.

Permissions

Who can view or edit a test is decided by the project's permission scheme.

Links

A test links to a story just like a bug links to a story.

Boards

Executions can appear on the agile board next to Stories and Bugs.

Dashboards

Jira gadgets read Xray data with no extra integration.

Result

Zero lost context. Requirement, test, and defect live in the same place.

Structural advantage over external TMS: being native issues, there is no syncing and no double source of truth.

The pieces · 3 of 18

Xray's 6 pieces

Six work-item types. Each one answers a different question.

Test

The WHAT. The specification of a test case.

Precondition

The BEFORE. A reusable initial state.

Test Set

The GROUP. A logical folder to organize tests.

Test Plan

The TRACKING. The umbrella that consolidates progress.

Test Execution

The WHEN. The task of running tests and collecting results.

Sub-Test Exec

The SAME, but a child of a requirement, for the board.

Mnemonic: each piece answers a question → WHAT / BEFORE / GROUP / TRACKING / WHEN. Source: docs.getxray.app — purpose of each Xray entity.

Reuse and group

The pieces · 7 of 18

Test Execution  and the Test Run

Test Execution

Assignable task of running a group of tests, for one version and one environment. It contains several Test Runs.

Test Run

Record of ONE run of ONE test. This is where status, evidence, defects, and per-step status live.

States of a run

TODO EXECUTING PASS FAIL ABORTED
Execution = the run session; Run = one line inside. States: TODO/EXECUTING/PASS/FAIL/ABORTED. Source: docs.getxray.app — Execute Tests.

The flow · 9 of 18

Xray's natural flow

① SPECIFY ② ORGANIZE ③ PLAN Test (case) → Test Set + → Test Plan Precondition (scope) │ │ └─────────────────┐ ▼ ④ EXECUTE ───────► Test Execution │ (Test Runs) ▼ ⑤ REPORT ────────► PASS / FAIL + 🐞 Defects │ ▼ ⑥ COVER ─────────► Coverage (OK / NOK)

Mnemonic · "SOPERC"

Specify · Organize · Plan · Execute · Report · Cover. Each test is a function that steps on stage, performs, and leaves a review.

A logical order, not a rigid one: an execution can be born ad hoc or cloned from a previous Plan. Source: docs.getxray.app — Process / Executing Tests.

How the pieces connect

STORY CALC-10 "Pay with card" │ covered by / "is tested by" ▼ TEST CALC-201 ─┐ TEST CALC-202 ─┤── grouped in ────► TEST SET "Payments" TEST CALC-205 ─┘ │ included in the scope ▼ TEST PLAN CALC-300 "Sprint 14·Payments" │ generates run cycles ▼ TEST EXECUTION CALC-400 ── contains ──► Test Runs │ ▼ when a run fails BUG CALC-455

A Story is covered by Tests → they enter a Plan → the Plan generates Executions → each Run can give birth to a Bug.

The backbone. The Story is a normal Jira issue (a "coverable requirement"). The rest are Xray issues. Source: docs.getxray.app — Requirement Traceability Report.

Coverage · 11 of 18

Is the requirement tested?

Xray computes a "Requirement Status" field on each Story from the status of its tests.

OK

All tests passed.

NOK

At least one failed.

NOT RUN

Some not run (no failures).

UNCOVERED

No tests associated.

Mnemonic · "the traffic light + grey"

🟢 OK · 🔴 NOK · 🟡 NOT RUN · ⚪ UNCOVERED. Coverage rolls up the hierarchy: if a child is NOK, the parent (Epic/Story) becomes NOK.

Calculation: all PASS→OK; any FAIL→NOK; any TODO with no fails→NOT RUN; no tests→UNCOVERED. Up to 3 levels (Epic→Stories→Subtasks). Source: docs.getxray.app — calculation of coverage status.

Traceability · 12 of 18

Traceability: two layers

Layer 1 · Jira links

The visible kinship

Jira's issue links: "tests", "is tested by". They connect Story ↔ test/plan/execution.

Layer 2 · Xray-internal

The DNA of the run

Xray's own membership: which Tests are in a Plan, which Runs in an Execution, which Defects in a Run. These are not issue links.

Mnemonic: "Jira links ISSUES, Xray links RESULTS." That is why something "looks linked" in Jira yet the run does not appear in the Plan — they are two different layers.

Critical for the repo's fix-traceability: in jira-xray you must repair BOTH layers. Layer 1 = acli/issue links; layer 2 = xray-cli (membership Test↔Plan↔Execution↔Run).

Defects · 14 of 18

Defects in Xray

The bug is born in the run. When a test fails, the tester creates it from the execution and it stays hooked to that Test Run.

  • It is a normal Jira issue of type Bug.
  • It links to the Test Run (not to the abstract Test).
  • It surfaces in the Defects column of the report.

Defect JQL in Xray

// bugs while executing a requirement issue in defectsCreatedForRequirement("CALC-10") // bugs during an execution issue in defectsCreatedDuringTestExecution(CALC-400)

Xray knows which bug came out of which run → bidirectional trace test ↔ defect.

"The bug is a child of the Run, not of the Test": the same Test can have 0 bugs in one run and 1 in another. Source: docs.getxray.app — Execute Tests, defectsCreatedForRequirement / defectsCreatedDuringTestExecution.

To take away · 17 of 18

Five ideas to not forget

1 · Everything is an issue. Xray lives inside Jira; it inherits JQL, permissions, and links.

2 · SOPERC. Specify·Organize·Plan·Execute·Report·Cover.

3 · Plan observes, Execution runs. The Run is the real unit of result.

4 · Traffic light + grey. OK·NOK·NOT RUN·UNCOVERED = coverage.

5 · Two trace layers. Jira links issues; Xray links results. The bug is a child of the Run.

In this repo: ATP→Plan · TC→Test · ATR→Execution. Plan first, always.
