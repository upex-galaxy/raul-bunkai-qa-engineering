PART 01

The Craft
Sprint QA by Hand

If the AI did not exist, what would a senior QA engineer do, step by step, to test one ticket well? Three stages, always in order: Planning → Execution → Reporting.

Imagine no AI. You are the QA engineer. This is the discipline.

the pipeline

One pipeline, every ticket

1

Session Start

Fetch ticket, load context, build the PBI folder, explain the story, WAIT for OK.

2

Stage 1 — Planning

Triage, then build the ATP (Acceptance Test Plan).

3

Stage 2 — Execution

Smoke first, then the UI/API/DB triforce.

4

Stage 3 — Reporting

ATR, QA comment, transition the ticket.

5

Hand off

Stages 4 docs, 5 automation, 6 regression — NOT this loop.

This order never changes. Session Start is always first; reporting always closes. You cannot smoke-test before a plan, or report before you execute. Stages 4-6 are other people's jobs.

pick the mode first

Three ways in, same three stages

Single — User Story

In: one story ID at Ready-for-QA. Out: ATP + ATR + QA comment, ticket moved to QA-approved. TC artifacts depend on modality (see "TC creation timing").

story run

Single — Bug

In: one bug ID on staging. Out: a triage decision, then code-review-only OR ATP + ATR + verification (no TCs in-sprint; regression Test reused/created in Stage 4 if worthy).

bug run

Batch sprint

In: a SPRINT-N-TESTING.md framework. Out: per-ticket artifacts + updated framework + summary, with resume support.

batch
Rule of thumb: a story at Ready-for-QA is a story run; a deployed bug is a bug run; "process sprint 5" is batch. The stages are identical — only bookkeeping differs.

session start — the universal entry

Before you test anything, set the table

  • Reachability preflight. curl WEB_URL & API_URL; a 2xx/3xx (login redirect counts) passes. 404/410/5xx or a dead deploy = STOP.
  • Fetch the ticket. Title, ACs, priority, comments — read them, do not skim.
  • Load project context. Business data/feature/API maps + master test plan.
  • Build the PBI folder. One per ticket, never reused.
  • Explain the story and WAIT for OK. Non-negotiable.
Session Start is a gate, not a formality. The reachability preflight alone prevents the highest-cost waste: authoring a full plan against a dead environment.

the gate that saves the run

Reachability ≠ smoke — both run, different questions

Reachability preflight

Asks "is the environment even up / can we receive the email?" — a cheap curl before any plan is written.

Stage 2 smoke test

Asks "does the feature actually work?" — only after the plan exists.

Watch out

A dead env or a send-only inbox (sends mail but can't receive a magic link) is caught HERE with a STOP — not after an hour authoring an ATP. For email/magic-link/auth stories, confirm the mailbox can actually receive.

A real, specific rule. Two different gates, two different questions. Skipping the first means false-positive bugs against a broken env.

the hand-off gate

Explain the story, then STOP and wait

Write a plain-language Story Explanation

What the feature is, how it works, what you will test, and any team decisions pulled from the ticket comments (scope changes, dev notes, PO clarifications).

Then WAIT for explicit confirmation

Never auto-advance Ready-for-QA → In Testing without it. Session Start is a hand-off, not a one-shot — the human stays in control of what gets tested.

Anti-pattern S10: never transition a ticket without explaining the story and waiting. The human stays in control of what gets tested.

STAGE 01

Planning
Decide What Deserves a Test

Not every ticket earns a full plan. Triage first: veto, then risk score. Then build the ATP — spec-driven, traceable, with realistic counts.

Planning is where you decide effort. Over-testing a CSS fix is as wrong as under-testing a payment flow.

spec-driven testing — the 4 pillars

Test from the spec, not from intuition

❌ From intuition

  • "I'll test the login and see what I find"
  • "Bug: the button doesn't work"
  • "I tested everything I could think of"
  • Random clicking through the app
→

✅ Anchored, not AC-bounded

  • Start from STORY-XXX's ACs, then probe beyond them
  • AC-3 of STORY-XXX fails: submit doesn't respond after click
  • Each AC verified (the floor) + boundaries / errors / states it's silent on
  • Charter-driven exploration on the story's risk areas
Spec-driven means ANCHORED (every case traces to an AC or a risk), not AC-BOUNDED. Every finding references the story AND the specific AC — but coverage doesn't stop when the ACs go green.

test-design doctrine — the core principle

Verifying acceptance criteria is the floor, not the testing

1 · Passing ≠ probing

An AC-pass confirms the business condition holds. It does not prove the system is robust, safe, or error-resistant.

2 · Floor, not ceiling

ACs are the mandatory baseline for "done". Quality testing exceeds that line — it never rests on it.

3 · Criterion vs case

A criterion is a business assertion. A test case is a concrete exploration of how that assertion holds or breaks.

4 · 1 : N by default

One AC → many cases (partitions, boundaries, states). Collapse to one only with a written "trivially atomic" reason.

5 · Risk lives outside

Defects cluster where the AC is silent: limits, invalid data, concurrency, timeouts, rollback, anomalies.

Coverage =

AC-conformance (floor) + risk-beyond-AC. "100% of ACs verified" is the floor — report it as such, then report what you probed above it.

This is the doctrine the whole ATP is built on. Canon: agentic-qa-core/references/test-design-doctrine.md. The Phase 4 outline set must cross both axes, never just the AC list.

EP without BVA misses off-by-one. A status field with no invalid-transition case is an incomplete plan. The triggers are binding — when one fires, the technique is required, not optional.

triage — the veto table

Veto beats risk score, always

VerdictTriggers
SKIP — code review onlypure text/label, CSS/visual, docs, config/infra, tech-debt refactor with no behavior change (text from API/DB is NOT pure text)
REQUIRE — force full testmoney/billing, data integrity on core entities, auth/authorization, external integrations, state-machine bugs, calculations/formulas
The veto runs before any scoring. A typo hardcoded in the frontend skips. Anything touching money, data, or auth gets tested regardless of how trivial the diff looks.

triage — the risk score

No veto? Score it, then act on the band

FactorPts
New feature (vs modification)+3
Dynamic data — API/DB (not hardcoded)+3
Explicit ACs present+2
User-facing / visible behavior+2
High effort (>4 person-hours)+2
High or Critical priority+1
Multi-component (multiple code areas)+1
These are the real weights. New + dynamic data is already +6 — most genuine features land MEDIUM or HIGH.

triage — the bands

0-3 LOW · 4-7 MEDIUM · 8+ HIGH

ScoreLevelAction
0-3LOWCode review only (treat as SKIP)

Present the triage result to the user before proceeding. MEDIUM is the default working band; HIGH adds extended edge cases.

building the atp

From acceptance criteria to executable outlines

  • Phase 1 — Critical Analysis. Anchor to business + technical context, rate complexity (logic/integration/data/UI).
  • Phase 2 — Story Quality. Surface ambiguities, gaps, edge cases; mark inferred behavior NEEDS PO/DEV CONFIRMATION.
  • Phase 3 — Refined ACs. Rewrite each as Given/When/Then with SPECIFIC data.
  • Phase 4 — Test Outlines. Name them "Should <BEHAVIOR> <CONDITION>", add parametrization tables.
  • Phase 0.3 — Data feasibility. Each AC is Discover / Modify / Generate, or blocked.
The ATP is the spec for your testing. Phases run in order; Phase 2's "no issues found" is a perfectly good answer — never invent questions to fill a checklist.

outline naming convention

"Should <BEHAVIOR> <CONDITION>" — name both

❌ Vague

  • Login test
  • Login - error
  • Test the form
  • Negative case
→

✅ Behavior + condition

  • Should login successfully with valid credentials
  • Should display authentication error when password is incorrect
  • Should accept character limit when entering exactly 50 chars
  • Should handle cart when there are multiple same items
BEHAVIOR = verb + object; CONDITION = the context that makes the case unique. Do NOT add a TC# prefix here — that is Stage 4's job. Never force a minimum count: a 1-line AC may need just 1 outline.

two atp homes

The ATP lives in Jira — the local file is a read-only cache

Modality jira-native

Write the ATP to the Story's acceptance_test_plan custom field (or a "## Acceptance Test Plan (ATP)" fallback comment if the field is absent).

Modality jira-xray

Write it to a Test Plan issue's description, linked "tests" → Story, with a paired Test Execution (the ATR container).

Watch out

Run the sync to materialize a local .md — but NEVER hand-write it. Jira is the source of truth (anti-pattern S14).

This split matters. Hand-authoring the synced cache gets overwritten and breaks the audit trail. Author in Jira, sync down, read back to confirm.

tc creation timing — modality-aware

When does a test case become a work item?

StageModality jira-nativeModality jira-xray (bun xray)
Stage 1 — PlanningTC outlines only (names + 1-line precond/expected). NO Test work items — a native Test issue is documentation.Create + execute Test issues for the planned outlines, run via a Test Execution. The Test IS Xray's execution unit, by plugin design.
Stage 2 — ExecutionRun outlines + explore beyond them; track PASS/FAIL in session memory.Execute the created Tests + explore beyond them. A probe becomes a Test only if it found a defect or is worth repeating.
Stage 4 — docsCreate Tests only for regression-worthy scenarios after ROI.Select + promote the regression-worthy Tests into the Regression Test Plan + Test Set, then enrich.

The principle

An execution artifact (how you run a test this sprint) is NOT the regression repository (the curated, repeatable set). The persistent regression set is ROI-gated in Stage 4 — never assumed in Stage 1.

This is the single rule that decides the whole skill's shape. The wide 1:N technique derivation feeds outlines (native) or sprint Test artifacts (xray) — either way it never auto-populates the regression repository. Heavy specification is spent only on Stage-4 regression candidates, never on Deferred scenarios.

stage 1 closeout

Verify traceability before you touch a browser

Run the trace check

Story → ATP → ATR must link, and each TC → Story + ATP + ATR. Add the label shift-left-reviewed and append the refined ACs to the ticket description.

Bugs are the exception

For bugs: ATP + ATR but NO TCs in-sprint. The bug ticket IS the immediate retest case; reproduction steps ARE the test steps — so "missing TC" trace warnings are expected. Golden rule: if the bug is regression-worthy, Stage 4 MUST end with a persistent Test — reuse the failed Test or create one. Not every bug qualifies (a one-time typo in a stable area is treated like a failed test).

Traceability is pillar #2 of SDT. If the chain is broken, the finding cannot survive an audit. Bugs are the deliberate in-sprint exception — but the regression golden rule still applies: a regression-worthy bug ends with a persistent Test in Stage 4 (reuse or create).

STAGE 02

Execution
Smoke First, Then the Triforce

Validate the environment before you validate the feature. Then explore through up to three layers — UI, API, DB — in the order the feature type demands.

Execution is where bugs actually surface. Order is everything: smoke before deep-dive, env before feature.

the smoke test — mandatory, always first

A 5-10 minute Go / No-Go gate

  • Basic access. App loads, no 500, assets render, console has no red (yellow warnings OK).
  • Authentication. Login, session persists on refresh, logout works (skip if pre-auth).
  • Happy path. Run the ticket's primary 3-5 step flow end to end.
  • Backend integration. Watch the Network tab; all /api/* calls 2xx, data persists after F5.
  • Verdict. PASS continues to deep exploration; FAIL STOPS the run.
Smoke validates the env; the triforce validates the feature. No edge cases, no minor bugs, no more than 10 minutes. A smoke FAIL is an env-level blocker — stop and report (anti-pattern S7: never skip smoke).

the triforce model

Pick the layer order by feature type

Feature typeOrder
UI-focusedUI → API → DB
API-firstAPI → DB → UI (if any)
Data-focusedDB → API → UI (if any)
Full-stackAll three
Triforce = up to three layers of validation. A UI click that fires a non-2xx escalates you to the API layer; data that looks wrong on screen sends you to the DB to see what actually persisted.

inside the triforce

What each layer actually checks

UI

Navigate, snapshot the DOM, run happy path, watch console + network, screenshot at the FAILING state. Edges: empty, max length, 0/-1/MAX_INT, <script>, '; DROP TABLE, refresh mid-flow, back button.

layer 1

API

Status-code matrix — 200 happy, 400 missing field, 401 no auth, 403 other-tenant, 409 duplicate. Replay with valid/invalid/boundary payloads.

layer 2

DB

Confirm rows persisted, triggers fired (stored total == SUM of items), constraint probes (FK/CHECK/UNIQUE/NOT NULL) wrapped in a transaction and ROLLBACK.

layer 3
Each layer has its own playbook. The API status-code matrix and the DB constraint probes are concrete checklists — you run them, you don't improvise them.

the rls probe — for multi-tenant apps

Two users, same role — can A see B's data?

With users A and B at the same role

(1) A lists resources → should return only A's rows; (2) A reads B's resource by id → empty array or 403; (3) A patches B's row → 0 rows affected or 403. Record each as VERIFIED or VULNERABLE.

Any VULNERABLE is Critical-severity

A cross-tenant read/write is security-exploitable and BLOCKING. Authorization bugs never show up in happy-path testing — you have to deliberately try to break tenant isolation.

One of the highest-value probes in the playbook. Authorization bugs do not surface in happy-path testing; you must deliberately try to break tenant isolation.

finding triage — graduated pause

A FAIL is not auto-Critical — triage, then stop or continue

ClassExamplesAction
BLOCKINGsmoke down, env down, data corruption, security-exploitable (auth bypass, cross-tenant, RLS VULNERABLE)STOP the pass immediately, surface to user
NON-BLOCKINGcosmetic, minor validation gap, edge-case on a non-critical TC, security/framework-default pending recalibrationlog it, mark TC FAILED, FINISH the remaining TCs, surface all at Stage 2 close
This rule protects coverage. Pausing a 17-TC pass on one cosmetic finding wastes the run and loses coverage; a genuine blocker must still stop instantly. Triage first.

don't leave coverage on the floor

Time-dependent cases get a decision path, never a bare BLOCKED

TTL / expiry / rotation cases

Magic links, OTPs, token expiry, sessions can't be tested by clicking faster — you need to control time. Pick the highest option the stack allows: (1) short-TTL test config/fixture — best; (2) clock-mock driving the real expiry logic; (3) explicit manual-defer with a written follow-up — last resort.

Record the decision

"BLOCKED — needs time fixture" with the option chosen and a one-line rationale. By Stage 2 close every TC is PASSED or FAILED — no NOT RUN.

A craft detail that separates a thorough tester from a lazy one. A bare "BLOCKED" silently loses coverage; the rule forces a documented decision.

STAGE 03

Reporting
Make the Verdict Auditable

Fill the ATR, triage every failure into a clean bug, post the right QA comment, and transition the ticket. The QA comment is a summary; the ATR is the audit record.

Reporting is where QA work becomes durable. A test you ran but didn't record didn't happen, as far as the audit is concerned.

bug severity matrix

Severity drives priority drives the title

SeverityCriteriaPriority
Criticalcore blocked, no workaround, data lossHighest
Majormain feature broken, workaround hardHigh
Moderatefeature issue with easy workaroundMedium
Minorlow-priority edge caseLow
Trivialcosmetic onlyLowest
Bug titles follow a fixed format: "<EPIC>: <COMPONENT>: <ISSUE_SUMMARY>" e.g. "CheckoutFlow: Payment: Error message not shown for incorrect password". Severity → priority is a lookup, not a judgment call.

filing a bug — the discipline

No bug ticket without a repro path AND evidence

  • Retest once to confirm. Reproduce via UI/API/DB before you believe it; "it failed once" is not a bug (S12).
  • Get human agreement. Present the full draft (title, severity, error type, environment, evidence) and WAIT for OK.
  • Search for duplicates. Using the EPIC:COMPONENT:SUMMARY pattern before filing.
  • One layer per ticket. Never mix UI + API + DB findings in one bug (S8); file per layer or root-cause cluster.
  • Attach the failing state. Screenshot/trace/log/HAR/DB row — capture the bug, not the navigation to it.
Every bug carries Steps to Reproduce, Technical Analysis, Impact, severity, error type, root cause, and labels (always bug + exploratory-testing). The human confirmation gate is mandatory — QA never files unilaterally.

the severity recalibration gate

A security FAIL might be a framework default — don't over-block

Run the gate before a blocking transition

On a security/auth/framework-default FAIL (cookie flags, CSP/HSTS, SDK-by-design): (1) state the mitigation hypothesis ("Secure/HttpOnly are framework defaults under HSTS preload"); (2) cite ONE verification fact (a config line, an observed header, a doc reference); (3) surface to the user.

Confirmed non-defect → GO-with-debt

Mark "PASSED WITH ISSUES", file a low-priority follow-up, do NOT block. QA never silently downgrades a security finding.

The Stage-3 counterpart of the bug veto tree. It stops QA from burning a fix cycle over a framework default — while keeping a human in the loop so a real vuln is never hidden.

close the ticket — the right way

ATR, comment template, then transition

OutcomeQA commentTransition
Story PASSEDTemplate A→ qa_approved (via qa_sign_off)
Story FAILED (confirmed)Template B + file bug→ blocked, or stay in_test + linked bug
Bug VERIFIEDTemplate C→ closed (via retest_passed)
Bug NOT FIXEDTemplate Dstay ready_for_qa, tag dev
Four templates cover two ticket types × two outcomes. Always prepare the comment BEFORE transitioning. The ATR result is PASSED / FAILED / PASSED WITH ISSUES — mixed results use the last one and still hand off passing TCs to Stage 4.

the non-negotiables

Anti-patterns a senior QA never commits

  • Never report without an ATR snapshot (S3). The comment summarizes; the ATR is the audit record.
  • Never skip smoke before the triforce (S7). A broken env produces false-positive bugs.
  • Never reuse a PBI folder across tickets (S9). Cross-contamination breaks evidence + traceability.
  • Never bypass the bug-triage tree (S5). Every failure gets veto → risk-score → severity + root cause.
  • Never mark a story approved without sign-off + a signed ATR (S1). All artifacts in English, evidence gitignored.
These anti-patterns are the guardrails. Each one maps to a real way QA work goes wrong. Internalize them and the rest of the method follows.

PART 02

The Acceleration
/sprint-testing

Everything in Part 1, driven from a terminal. One trigger phrase, auto-loaded context, sequential subagents, and a human gate at every checkpoint.

Now the payoff: the skill that makes the AI run this entire craft for you — while you stay in control.

how you invoke it

Trigger phrase + what auto-loads

Triggers — natural language, no slash

"test this ticket" · "QA this user story" · "retest bug PROJ-445" · "verify this fix" · "run exploratory testing" · "process sprint 5" · "next ticket in sprint" · "resume sprint testing".

describe the task

Auto-loaded on Session Start

project.yaml (identity, env URLs, MCP names) · jira-required.yaml + jira-fields.json + jira-workflows.json · business data/feature/api maps · master-test-plan.md · the ticket via jira:sync-issues · .env · kata-manifest.json.

You don't paste instructions. You describe the task; the skill resolves the mode and pulls every context file the methodology needs, in a fixed reading order.

how the human stays in control

Confirmations and gates are built in, not optional

  • Plan before code. Session Start STOPS for your OK on the story explanation before any plan is written.
  • Triage is presented. Veto/risk-score result shown before a single test runs; MEDIUM band asks you.
  • Bugs need a yes. Full draft surfaced, human confirmation gate before any bug is created.
  • Blocking findings pause; tool failures STOP. No auto-fix, no auto-rollback; you choose retry/skip/abort.
  • Resume, never restart blind. progress.md lets you pick up exactly where a session was interrupted.
The acceleration is real, but the methodology's human checkpoints are preserved one-for-one. The AI does the typing; you keep the judgment calls.
