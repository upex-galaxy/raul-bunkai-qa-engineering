Welcome the room. This is the bridge between manual QA and automation — the discipline that decides what we actually invest in automating, and how we keep it auditable.

Why this discipline exists

< 0.5

Most tests should never be automated

ROI below 0.5 = Defer. Every test has a maintenance cost, most bugs don't recur, and fewer well-chosen tests beat many low-value ones. If a module yields 80 scenarios and 60 land in regression, you scored too softly — re-apply the filter harder.

Anchor the whole talk on this counter-intuitive truth: the default answer is Defer. Automation is an investment, not a reflex.

PART 01

The Craft
documenting tests by hand

If the AI did not exist, what would a senior QA engineer do — step by step — to document tests well? Every stage, every gate, every formula.

For this whole section, imagine no automation. Just you, Jira, and a methodology you have to apply with judgment.

The one hard prerequisite

Only document what's already validated

The TMS is for documentation + regression protection — NOT exploration

You may only document behavior already validated: a QA-Approved story, a closed/verified bug, or a finished exploratory session. If you can't confidently describe the expected behavior, the feature isn't ready for the TMS.

Explore → Analyze → Document. Never document mid-exploration.

This is the gate before the gate. Documenting unstable behavior creates maintenance debt. If you're still finding out what the system does, you're not ready to write a TC.

The spine of the method

Three phases, always in this order

1 · Analyze

Identify the real scenarios; separate user flows from cross-cutting traits; validate the design against actual code.

find scenarios

2 · Prioritize (ROI)

Run every scenario through 3 filters, then the ROI formula → Candidate / Manual / Deferred.

the gate

3 · Document

Create ATP/ATR/TC in the TMS, link them, transition states, cache locally.

write + link

Never skip Prioritize

Most scenarios should end Deferred. The order is non-negotiable.

Memorize this spine. Skipping prioritization is how teams end up with bloated, brittle suites. The order is non-negotiable.

Gate 0 — pick your TMS modality

Same concepts, different containers

Modality A — Xray on Jira

  • ATP = a real Test Plan issue
  • ATR = a real Test Execution holding Test Runs
  • TC = an Xray Test (Manual/Cucumber/Generic)
  • CI imports JUnit/Cucumber → runs auto-update
  • Test Set / Pre-Condition are first-class
vs

Modality B — Jira-native, no Xray

  • ATP = Story's Acceptance Test Plan field
  • ATR = Story's Acceptance Test Results field
  • TC = a custom Test issue type
  • no separate plan/execution issues
  • CI updates a Test Status field via script
Before anything else, answer: does this project have Xray licensed? The artifacts are the same four concepts but live in completely different containers. Resolve once, treat as sticky, never mix the two inside one story. Neither column is "bad" — they're the two valid worlds.

Resolving modality without asking

Auto-resolve before you interrupt the user

1

Check config

Look for the TMS CLI value: bun xray → Xray; acli-only/unset → Jira-native.

2

Check the MTP

A line like TMS: Xray on Jira or TMS: Jira native.

3

List issue types

If Test Plan / Test Execution / Test Set / Pre-Condition exist → Xray; else native.

4

Only then ask

If all three checks fail, ask the one question. Persist; don't re-resolve mid-session.

A good engineer doesn't interrupt with questions they can answer themselves. Three deterministic probes first; the human is the last resort, not the first.

PART 01 · PHASE 1

Analyze
find the REAL scenarios

Before you score or document anything, you have to know what a test case actually is — and what is not one.

Analysis is where most beginners go wrong. They count assertions as tests and treat cross-cutting traits as scenarios. Let's fix that.

What is NOT a test case

Cross-cutting traits live inside tests

❌ NOT standalone TCs

  • Mobile responsive
  • XSS prevention
  • Performance
  • Accessibility
  • API contract / generic error handling
→

✓ How they're actually validated

  • run each test in a mobile viewport
  • special-character test data inside tests
  • timing assertions inside tests
  • a11y assertions inside UI tests
  • schema checks / specific negative paths
These are APP-LEVEL suites, not per-feature TCs. Promoting "XSS prevention" to its own test case is a classic beginner mistake — it's a property you assert everywhere, not a flow you test once.

The load-bearing rule of Analyze

One (Precondition + Action) = one TC

A TC is defined by Precondition + Action

ALL expected results from that same pair belong to the SAME TC.

✓ SAME TC (5 assertions)

  • pre valid credentials, action submit login
  • redirect to dashboard + auth token
  • profile via /auth/me + session cookie + welcome msg
≠

✓ DIFFERENT TCs (preconditions differ)

  • valid credentials → success (TC-A)
  • locked account → 423 (TC-B)
  • invalid credentials → 401 (TC-C)
Burn this in. If the precondition and action are identical, it's one test case no matter how many things you assert. Different preconditions producing different outcomes — those are different TCs. Splitting "check panel A / B / C" is the single most-diagnosed anti-pattern in reviews.

test-design doctrine — the core principle

An AC→TC map is the floor, not the coverage

1 · Passing ≠ probing

"Every AC has a green TC" confirms the contract. It does not prove the system is robust, safe, or error-resistant.

2 · Floor, not ceiling

≥1 TC per AC is the minimum, never the target. Quality documentation exceeds the AC baseline.

3 · Criterion vs case

A criterion is a business assertion. A TC is a concrete exploration: precondition + action + verifiable outcome.

4 · 1 : N by default

One AC → many TCs (partitions, boundaries, states). Collapse to one only with a written "trivially atomic" reason.

5 · Risk lives outside

Document the boundary / negative / state / anomaly TCs the AC is silent on — that is where regressions hide.

Three layers, three counts

Derive many, document few, automate fewer. analyzed 80 → documented 12 → automated 8 — never "document all 80". ROI prunes; the rest are Deferred, never deleted.

Canon: agentic-qa-core/references/test-design-doctrine.md. ROI is an automation filter, not a derivation filter. The next slide makes the three-layer / derive-wide-document-narrow rule explicit.

derive wide · document narrow · automate fewer

Three layers, three counts — not one

1

DESIGN / derive (1:N)

In sprint-testing planning + exploration: consider many cases by technique. Lives in the prioritization analysis, not yet in the TMS.

2

DOCUMENT only the repeatable

Create a persistent TMS TC only for Candidate + Manual — scenarios worth re-running. jira-native: Deferred is NOT created in the TMS (report only). jira-xray: "document 12" means promote 12 into the Regression Test Plan, leaving the rest as unpromoted sprint artifacts.

3

AUTOMATE the Candidates

Only the Candidates flow to /test-automation. Manual and Deferred are terminal.

The healthy shape

analyzed 80 → documented 12 → automated 8. You document a test because it will be re-executed — never to hit a count. Most scenarios are Deferred.

This is the spine of the whole discipline. Deriving widely is free; persisting is ROI-gated. The single biggest beginner mistake is "document all 80" — treating documentation as a coverage scoreboard instead of a regression-protection investment. In native, Deferred never even becomes a Jira issue. In Xray the 80 may already be sprint Test artifacts, so "document 12" is really "promote 12". If most scenarios end up Candidate/Manual, re-apply Phase 0 harder.

Don't trust the plan — read the code

Source-code validation is mandatory

  • Why. The ATP was written BEFORE the code existed; its assumptions may be stale.
  • Open the implementation plan. List the files it touches.
  • Grep the real code. Search for data-testid=, route handlers, API paths, text formats.
  • Common discrepancies. Assumed API is actually SSR/direct-DB; "based on N reviews" vs real "(N reviews)"; hardcoded IDs vs a variable pattern.
  • Log it. Record divergences in a Refinement Notes section on the TC.
The plan is a hypothesis; the code is the truth. Five minutes of grepping saves a whole broken automation suite downstream. Always reconcile design against implementation before you document.

Naming + partitioning

Name a TC so anyone can find it

  • The format that matters. {US_ID or TS_ID}: TC#: Validate <CORE> <CONDITIONAL>
  • CORE. verb + object: successful login, authentication error, order creation.
  • CONDITIONAL. the distinguishing condition: with valid credentials, when password is incorrect.
  • Worked example. GX-101: TC2: Validate authentication error when password is incorrect.
  • Reject these. Login test · Login - error · TC1: Test form · Should work correctly.
A good name is searchable, unique, and self-documenting. Equivalence partitioning: same output → one parameterized TC (Scenario Outline + Examples); different output → separate TCs. The anti-patterns are real titles people ship — vague and useless six months later.

PART 01 · PHASE 2

Prioritize
the ROI gate

The heart of the discipline: deciding, with numbers, which validated scenarios are worth the lifetime cost of automation.

Now the judgment phase. Every scenario passes through filters, then a formula, then lands in exactly one of three buckets.

Phase 0 — apply BEFORE the formula

Three filter questions; fail any → Deferred

1

Future regressions?

Does it protect against FUTURE regressions? A one-time typo in a stable area → No → Defer.

2

Prior bugs here?

Yes → prioritize even at moderate ROI. "It failed once, it can fail again."

3

App- or feature-level?

XSS / a11y / perf / responsive are APP-LEVEL suites → Defer from this scope.

Gate logic

Phase 0 fails → Deferred immediately. Phase 0 passes → now compute ROI.

These three questions kill most candidates before you ever do arithmetic. The prior-bug rule is special — it can override a mediocre ROI score, because a bug is empirical proof the area regresses.

The formula you must memorize

÷

ROI = (Frequency × Impact × Stability) / (Effort × Dependencies)

Three multipliers on top (Frequency, Impact, Stability — higher is better). Two DIVISORS on the bottom (Effort, Dependencies — higher is WORSE). A "critical flow" with Effort=5 and Dependencies=5 SHOULD score low — that's the formula working, not a bug. Each factor scored 1-5 independently.

The divisors are the part people get wrong. Effort and Dependencies belong in the denominator. A glamorous, expensive, externally-coupled flow is correctly penalized — automation cost is real.

Scoring each factor 1-5

The ROI rubric

Factor135
Frequency (how often run)Yearly / rarelyEvery sprintEvery PR / commit
Impact (if it fails)CosmeticDegrades UXRevenue / core business
Stability (of the flow)Very volatileModerateUnchanged for months
Effort to automate (÷)Trivial1-2 daysWeek+
Dependencies (÷)None3-4Complex externals
Walk one example: a login flow run on every PR (5), revenue-critical (5), stable for months (5), hours to automate (Effort 2), one dependency (1). That's (5×5×5)/(2×1) = 62.5 — automate without hesitation. Most flows are not login.

The reusability multiplier

Component Value bonus — reuse rescues low ROI

Component Value = Base ROI × (1 + 0.2 × N)

N = number of E2E flows that reuse the TC.

A humble atomic like authenticateSuccessfully scores base ROI 1.5 — borderline Defer. Reused in 5 E2E flows: 1.5 × (1 + 0.2×5) = 1.5 × 2.0 = 3.0 → now an automate-worthy Candidate. Reusable components earn their keep through composition, not solo value.

This is why atomic, composable test cases matter. A login step nobody automates in isolation becomes essential once five flows depend on it. Think of TCs as reusable components.

The verdict — exactly one of three

No fourth bucket

If >50% land Candidate/Manual, re-apply Phase 0 harder. Most scenarios should be Deferred.

Three buckets, no exceptions. Deferred in jira-native doesn't even create a TMS issue — you just document the decision in the prioritization report. In jira-xray the sprint Test already exists from sprint-testing Stage 1, so Deferred there means it stays as an unpromoted sprint artifact, not deleted. If too much is surviving, your filter is too soft.

Same verdict, different action

Modality changes the verb, not the verdict

The ROI outcome (Candidate / Manual / Deferred) is identical in both worlds. What differs is what Phase 3 does with it.

jira-native — CREATE

  • No sprint Tests exist yet
  • Phase 3 creates a Test work item for Candidate + Manual only
  • Deferred = report-only, no issue created
  • Same Analyze→Prioritize→Document pipeline
vs

jira-xray — PROMOTE + ENRICH

  • The Tests already exist from sprint-testing Stage 1
  • Phase 3 selects + promotes the regression-worthy ones into the Regression Test Plan + Test Set
  • and enriches them (rich Gherkin, parameterization, edge cases)
  • Deferred sprint Tests stay unpromoted, not deleted
This is the single most-missed nuance. The verdict math is shared; the verb is modality-specific. In native you author Tests from scratch for the survivors. In Xray the survivors were already born as sprint Test artifacts in Stage 1 — your job is to pick the regression-worthy ones, add them to the Regression Test Plan and the feature Test Set, label them regression-candidate, and specify them much more richly. Deferred is never a deletion in either world.

Bug-driven — the rule worth gold

A regression-worthy bug MUST end with a Test

Not every bug becomes a regression TC — but if it does, it must end with a Test. Reuse the existing failed Test if the bug came from one; otherwise create one. Both modalities.

❌ Does NOT qualify → Deferred

  • one-time typo in a stable area
  • treated like a failed test — fix already verified in sprint-testing
  • no new Test, log as Deferred
→

✓ Regression-worthy → must have a Test

  • came from an existing failed Test → REUSE it, link + promote, never duplicate
  • no prior Test → CREATE + design one
  • link via tests / is tested by

Overrides "the bug is the test case"

That sprint-testing phrase covers only the immediate in-sprint retest — NOT future regression. This rule decides whether a persistent Test must exist so the bug can never silently return.

Golden rule: where an important bug exists, a test must catch it again. A closed bug is strong empirical evidence the area regresses, so most qualify and lean Candidate — but not all. The decision tree: regression candidate? No → treat as a failed test, Deferred, done. Yes → did it come from an existing executed Test? Reuse that one. No prior Test? Create and design one. Never reopen a closed bug to attach the TC — file a new TC and link it.

PART 01 · PHASE 3

Document
create & link in the TMS

The scenarios that survived get written into Jira/Xray as real, linked, traceable artifacts — in a strict order.

Documentation is mechanical but unforgiving about order. Get the sequence wrong and you create orphans that only a traceability-repair tool can fix.

The four core entities have one permanent home in the Regression Epic — never orphan a TC outside it. The Test Set is an Xray-only feature-level organizer; the next slide separates the three containers people tend to conflate.

Don't conflate the three containers

Epic · Test Set · Test Plan — three distinct things

Regression Epic

The repository umbrella — one per project. Every documented TC lives under it. type=Epic, labels=test-repository.

umbrella

Test Set xray only

Feature grouping, 1:1 with the Epic/feature. Holds the promoted regression Tests of ONE feature. Native equivalent = Regression Epic + a feature/Epic label.

feature

Test Plan

Execution / regression scope — the set of Tests run for a release or regression cycle. ATP in jira-xray is a Test Plan issue.

execution

Lazy auto-create the Test Set (ask first)

jira-xray: if the feature's Test Set is missing when a promotion needs it, the AI creates it lazily in Stage 4 — ask the user first, mirroring the Regression-Epic ask-before-create rule. Only promoted Candidate/Manual Tests are added; Deferred sprint Tests are not.

People routinely fuse these. The Epic is the whole repository. A Test Set groups one feature's regression Tests, one-to-one with the feature Epic. A Test Plan is an execution scope. In jira-native there is no Test Set entity at all — you achieve the same grouping with the Regression Epic plus a feature/Epic label on each TC. The AI only creates the Test Set lazily, when a promotion actually needs it, and always asks before creating.

The order is not optional

ATP → ATR → link → THEN every TC

Linking order

(1) Create ATP, link to US. (2) Create ATR, link to US. (3) Update ATP, link to ATR (bidirectional). (4) For each TC: create, then link to US + ATP + ATR + AC. (5) For each promoted TC — xray: add to the feature Test Set + Regression Test Plan + label regression-candidate; native: apply the feature/Epic label (no Test Set entity).

By the time a TC is created, both ATP and ATR already exist, so all three links land at creation. Create TCs first and you get orphans — and fix-traceability becomes the only way out. The model is bidirectional: US↔ATP↔ATR↔TC. Given any one, you must reach the other three — that IS traceability.

This is the order people break under deadline pressure. Plan and results first, then tests. Cardinality matters too: US↔ATP and US↔ATR are 1:1; an ATR is never linked to two ATPs.

The artifact — high-quality Gherkin

Variables, never hardcoded data

TC-GX-101.feature
@critical @regression @automation-candidate @GX-101
Scenario Outline: Validate authentication error when password is incorrect
  """
  Bugs covered: GX-202   Related Story: GX-101   ROI Score: 4.0
  """
  # === PRECONDITIONS ===
  Given a verified user exists with {email} in the database
  # === ACTION ===
  When the user submits login with {email} and {wrong_password}
  # === VALIDATIONS ===
  Then a 401 error is returned with format "<message>"
  # === EQUIVALENT PARTITIONS ===
  Examples: Wrong password | Empty password | Unknown email
Variables table is mandatory: {email} → SELECT email FROM users WHERE active=true LIMIT 1. Tags always carry priority + suite + automation flag + story ID. Never paste a literal UUID or email — staging data changes over years. Structured comments and docstring metadata make the TC self-explaining and automatable.

Workflow state machine — never skip a state

How a TC moves through its life

1

Draft → In Design → Ready

Documentation written and reviewed.

2

Ready splits

for_manual → Manual (terminal), OR automation_review → In Review.

3

In Review → Candidate

approve_to_automate → In Automation → Pull Request → Automated.

No jumps

Draft can't go straight to Automated. Rework only via back_from_<state>. Deprecated reachable from any state. Manual can re-enter In Review if ROI changes.

The lifecycle is strictly sequential. Remember two independent fields: Test Status (workflow — where it is in its life) versus Execution Status (the run — did it pass last time). "Automated" + last run "FAIL" is a normal, valid combination.

Finish the job cleanly

Labels, the two-call Xray rule, and the local cache

  • Scope label (required). regression almost always; smoke only for critical paths (10-20% of suite); plus e2e/integration/functional.
  • Status labels. automation-candidate ↔ manual-only are mutually exclusive; swap to automated once merged.
  • Xray needs TWO calls. one Create Test (registers) THEN one Update Issue for the Description — skip the second and the TC has no readable docs.
  • Never hand-write ADF JSON. use the markdown→ADF path; hand-authored ADF drifts and breaks renderers.
  • Cache locally. one markdown per TC under .../stories/STORY-<KEY>/test-cases/{TC-ID}.md for clean handoff.
These finishing details separate a clean repository from a messy one. The Xray two-call pattern is the most common silent failure — a registered test with an empty body.

PART 02

The Acceleration
driving it from the terminal

Everything in Part 1 — the modality gate, the ROI math, the linking order, the workflow transitions — the /test-documentation skill executes from one command. Here's how, and how you stay in control.

Now we map the manual craft onto the skill. Same rules, same gates — the AI just runs them faster and never forgets the linking order.

Trigger + what loads

One phrase, full context auto-loaded

  • Trigger. /test-documentation, or natural phrasing: "document tests", "ROI analysis", "which tests to automate", "Candidate vs Manual", "turn this bug into a regression test".
  • Phase -1 resume check. reads .session/test-documentation/<scope>/progress.md; offers resume / restart / abort if a prior run was interrupted.
  • Phase 0 modality gate. auto-resolves Xray vs Jira-native (config → master-test-plan → list issue types → ask).
  • Auto-loads context. the synced Story folder (story.md, acceptance-criteria.md, comments.md), the Jira catalogs, and master-test-plan.md.
You don't paste the methodology — the skill carries it. It even checks whether a previous documentation session was interrupted and offers to resume exactly where it stopped.

Orchestration model

Main thread decides; subagents do the bulk writes

Inline (orchestrator)

Phase 1 Analyze and Phase 2 ROI stay in the main thread; decisions and planning don't get delegated.

judgment

Parallel (subagents)

Phase 3 bulk TC creation when N > 10: shard into chunks of ~5-10, cap at 10 subagents (~10 writes/sec → avoid 429s).

fan-out

Inline again

Traceability linking and the final coverage matrix need aggregated state, so they run in the orchestrator after subagents return.

aggregate
The split is deliberate: judgment lives in the command center, raw API writes fan out. The 10-subagent cap exists because both Jira and Xray throttle around 10 writes per second.
