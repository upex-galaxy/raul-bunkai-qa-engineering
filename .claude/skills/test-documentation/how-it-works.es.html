Metodología QA · Etapa 4

Test Documentation y ROI
— El oficio detrás de la Terminal

Cómo un ingeniero QA convierte tests validados en un repositorio TMS trazable y priorizado — primero a mano, luego a velocidad de terminal.

Analizar → Priorizar → DocumentarJira / Xray · US↔ATP↔ATR↔TC
#qa #roi #traceability
Da la bienvenida a la sala. Este es el puente entre la QA manual y la automatización — la disciplina que decide qué invertimos realmente en automatizar, y cómo lo mantenemos auditable.

Por qué existe esta disciplina

< 0.5

La mayoría de los tests nunca deberían automatizarse

ROI por debajo de 0.5 = Deferred. Cada test tiene un costo de mantenimiento, la mayoría de los bugs no recurren, y pocos tests bien elegidos superan a muchos de bajo valor. Si un módulo arroja 80 escenarios y 60 llegan a regression, puntuaste con demasiada suavidad — reaplica el filtro con más rigor.

Ancla toda la charla en esta verdad contraintuitiva: la respuesta por defecto es Deferred. La automatización es una inversión, no un reflejo.

PARTE 01

El Oficio
documentar tests a mano

Si la IA no existiera, ¿qué haría un ingeniero QA senior — paso a paso — para documentar tests bien? Cada etapa, cada control, cada fórmula.

Para toda esta sección, imagina que no hay automatización. Solo tú, Jira y una metodología que debes aplicar con criterio.

El único prerrequisito firme

Solo documenta lo que ya está validado

El TMS es para documentación + protección de regression — NO para exploración

Solo puedes documentar comportamiento ya validado: una Story QA-Approved, un bug cerrado/verificado, o una sesión exploratoria terminada. Si no puedes describir con confianza el comportamiento esperado, la funcionalidad no está lista para el TMS.

Explorar → Analizar → Documentar. Nunca documentes a mitad de la exploración.

Este es el control antes del control. Documentar comportamiento inestable crea deuda de mantenimiento. Si todavía estás averiguando qué hace el sistema, no estás listo para escribir un TC.

Tres fases, siempre en este orden

el control

La mayoría de los escenarios deberían terminar en Deferred. El orden es innegociable.

Memoriza esta columna vertebral. Saltarse la priorización es como los equipos terminan con suites infladas y frágiles. El orden es innegociable.

Control 0 — elige tu modalidad de TMS

Mismos conceptos, distintos contenedores

Modalidad A — Xray sobre Jira

  • ATP = un issue Test Plan real
  • ATR = un Test Execution real que contiene Test Runs
  • TC = un Test de Xray (Manual/Cucumber/Generic)
  • CI importa JUnit/Cucumber → las runs se actualizan solas
  • Test Set / Pre-Condition son de primera clase
vs

Modalidad B — Jira nativo, sin Xray

  • ATP = campo Acceptance Test Plan de la Story
  • ATR = campo Acceptance Test Results de la Story
  • TC = un issue type Test personalizado
  • sin issues separados de plan/execution
  • CI actualiza un campo Test Status vía script
Antes que nada, responde: ¿este proyecto tiene Xray licenciado? Los artefactos son los mismos cuatro conceptos pero viven en contenedores completamente distintos. Resuélvelo una vez, trátalo como fijo, nunca mezcles los dos dentro de una Story. Ninguna columna es "mala" — son los dos mundos válidos.

Resolver la modalidad sin preguntar

Auto-resuelve antes de interrumpir al usuario

2

Revisa el MTP

Una línea como TMS: Xray on Jira o TMS: Jira native.

3

Si existen Test Plan / Test Execution / Test Set / Pre-Condition → Xray; si no, nativo.

4

Solo entonces pregunta

Si los tres chequeos fallan, haz la única pregunta. Persiste; no re-resuelvas a mitad de sesión.

Un buen ingeniero no interrumpe con preguntas que él mismo puede responder. Tres sondeos deterministas primero; el humano es el último recurso, no el primero.

PARTE 01 · FASE 1

Analizar
hallar los escenarios REALES

Antes de puntuar o documentar nada, debes saber qué es realmente un test case — y qué no lo es.

El análisis es donde la mayoría de los principiantes se equivocan. Cuentan aserciones como tests y tratan rasgos transversales como escenarios. Vamos a corregirlo.

Qué NO es un test case

Los rasgos transversales viven dentro de los tests

❌ NO son TC independientes

  • Responsive móvil
  • Prevención de XSS
  • Rendimiento
  • Accesibilidad
  • Contrato de API / manejo genérico de errores
→

✓ Cómo se validan realmente

  • ejecuta cada test en un viewport móvil
  • datos de test con caracteres especiales dentro de los tests
  • aserciones de tiempo dentro de los tests
  • aserciones de a11y dentro de los tests de UI
  • chequeos de schema / rutas negativas específicas
Estas son suites A NIVEL DE APP, no TC por feature. Promover "prevención de XSS" a su propio test case es un error clásico de principiante — es una propiedad que aseveras en todas partes, no un flujo que pruebas una vez.

La regla portante de Analizar

Una (Precondición + Acción) = un TC

Un TC se define por Precondición + Acción

TODOS los resultados esperados de ese mismo par pertenecen al MISMO TC.

✓ MISMO TC (5 aserciones)

  • pre credenciales válidas, acción enviar login
  • redirección al dashboard + token de auth
  • perfil vía /auth/me + cookie de sesión + mensaje de bienvenida
≠

✓ TC DISTINTOS (las precondiciones difieren)

  • credenciales válidas → éxito (TC-A)
  • cuenta bloqueada → 423 (TC-B)
  • credenciales inválidas → 401 (TC-C)
Graba esto a fuego. Si la precondición y la acción son idénticas, es un solo test case sin importar cuántas cosas asseveres. Precondiciones distintas que producen resultados distintos — esos son TC distintos. Dividir "verifica el panel A / B / C" es el antipatrón más diagnosticado en las revisiones.

doctrina de diseño de pruebas — el principio central

Un mapa AC→TC es el piso, no la cobertura

1 · Aprobar ≠ probar

"Cada AC tiene un TC en verde" confirma el contrato. No prueba que el sistema sea robusto, seguro ni resistente al error.

2 · Piso, no techo

≥1 TC por AC es el mínimo, nunca la meta. La documentación de calidad supera la línea base del AC.

3 · Criterio vs caso

Un criterio es una afirmación de negocio. Un TC es una exploración concreta: precondición + acción + resultado verificable.

4 · 1 : N por defecto

Un AC → muchos TCs (particiones, límites, estados). Colapsar a uno solo con justificación escrita "trivialmente atómico".

5 · El riesgo vive fuera

Documenta los TCs de límite / negativos / estado / anomalía que el AC calla — ahí se esconden las regresiones.

Tres capas, tres conteos

Deriva muchos, documenta pocos, automatiza menos. analizados 80 → documentados 12 → automatizados 8 — nunca "documentar los 80". ROI poda; el resto son Deferred, nunca borrados.

Canon: agentic-qa-core/references/test-design-doctrine.md. ROI es un filtro de automatización, no de derivación. La siguiente diapositiva hace explícita la regla de tres capas / derivar-amplio-documentar-estrecho.

derivar amplio · documentar estrecho · automatizar menos

Tres capas, tres conteos — no uno

1

DISEÑAR / derivar (1:N)

En la planificación + exploración de sprint-testing: considera muchos casos por técnica. Vive en el análisis de priorización, aún no en el TMS.

2

DOCUMENTAR solo lo repetible

Crea un TC persistente en el TMS solo para Candidate + Manual — escenarios que vale la pena reejecutar. jira-native: Deferred NO se crea en el TMS (solo reporte). jira-xray: "documentar 12" significa promover 12 al Regression Test Plan, dejando el resto como artefactos de sprint sin promover.

3

AUTOMATIZAR los Candidates

Solo los Candidates fluyen a /test-automation. Manual y Deferred son terminales.

La forma saludable

analizados 80 → documentados 12 → automatizados 8. Documentas un test porque será reejecutado — nunca para alcanzar un conteo. La mayoría de los escenarios son Deferred.

Esta es la columna vertebral de toda la disciplina. Derivar amplio es gratis; persistir está controlado por ROI. El mayor error de principiante es "documentar los 80" — tratar la documentación como un marcador de cobertura en vez de una inversión de protección de regression. En nativo, Deferred ni siquiera se vuelve un issue de Jira. En Xray los 80 ya pueden ser artefactos Test de sprint, así que "documentar 12" en realidad es "promover 12". Si la mayoría de los escenarios terminan en Candidate/Manual, reaplica la Fase 0 con más rigor.

No confíes en el plan — lee el código

La validación contra el código fuente es obligatoria

  • Por qué. El ATP se escribió ANTES de que el código existiera; sus supuestos pueden estar obsoletos.
  • Abre el implementation plan. Lista los archivos que toca.
  • Grep al código real. Busca data-testid=, route handlers, rutas de API, formatos de texto.
  • Discrepancias comunes. Una API asumida en realidad es SSR/DB-directa; "based on N reviews" vs el real "(N reviews)"; IDs hardcodeados vs un patrón de variable.
  • Regístralo. Anota las divergencias en una sección Refinement Notes del TC.
El plan es una hipótesis; el código es la verdad. Cinco minutos de grep ahorran una suite de automatización entera rota más adelante. Siempre reconcilia el diseño contra la implementación antes de documentar.

Nomenclatura + particionamiento

Nombra un TC para que cualquiera lo encuentre

  • El formato que importa. {US_ID or TS_ID}: TC#: Validate <CORE> <CONDITIONAL>
  • CORE. verbo + objeto: successful login, authentication error, order creation.
  • CONDITIONAL. la condición que lo distingue: with valid credentials, when password is incorrect.
  • Ejemplo resuelto. GX-101: TC2: Validate authentication error when password is incorrect.
  • Rechaza estos. Login test · Login - error · TC1: Test form · Should work correctly.
Un buen nombre es buscable, único y autoexplicativo. Particionamiento por equivalencia: misma salida → un TC parametrizado (Scenario Outline + Examples); distinta salida → TC separados. Los antipatrones son títulos reales que la gente entrega — vagos e inútiles seis meses después.

PARTE 01 · FASE 2

Priorizar
el control de ROI

El corazón de la disciplina: decidir, con números, qué escenarios validados valen el costo de por vida de la automatización.

Ahora la fase de criterio. Cada escenario pasa por filtros, luego una fórmula, y aterriza en exactamente uno de tres buckets.

Fase 0 — aplica ANTES de la fórmula

Tres preguntas filtro; falla cualquiera → Deferred

1

¿Regressions futuras?

¿Protege contra regressions FUTURAS? Un typo de una sola vez en un área estable → No → Deferred.

2

¿Bugs previos aquí?

Sí → prioriza aun con ROI moderado. "Falló una vez, puede fallar de nuevo."

3

¿Nivel de app o de feature?

XSS / a11y / perf / responsive son suites a NIVEL DE APP → Deferred desde este alcance.

Lógica del control

Fase 0 falla → Deferred de inmediato. Fase 0 pasa → ahora calcula el ROI.

Estas tres preguntas matan a la mayoría de los candidatos antes de hacer aritmética. La regla del bug previo es especial — puede anular un puntaje de ROI mediocre, porque un bug es prueba empírica de que el área regresa.

La fórmula que debes memorizar

÷

ROI = (Frequency × Impact × Stability) / (Effort × Dependencies)

Tres multiplicadores arriba (Frequency, Impact, Stability — más alto es mejor). Dos DIVISORES abajo (Effort, Dependencies — más alto es PEOR). Un "flujo crítico" con Effort=5 y Dependencies=5 DEBERÍA puntuar bajo — esa es la fórmula funcionando, no un bug. Cada factor se puntúa de 1 a 5 de forma independiente.

Los divisores son la parte que la gente confunde. Effort y Dependencies van en el denominador. Un flujo glamoroso, caro y acoplado externamente se penaliza correctamente — el costo de automatización es real.

Puntuando cada factor de 1 a 5

Recorre un ejemplo: un flujo de login que corre en cada PR (5), crítico para ingresos (5), estable por meses (5), horas para automatizar (Effort 2), una dependencia (1). Eso es (5×5×5)/(2×1) = 62.5 — automatiza sin dudar. La mayoría de los flujos no son login.

El multiplicador de reutilización

Bonus de Component Value — la reutilización rescata un ROI bajo

Component Value = Base ROI × (1 + 0.2 × N)

N = número de flujos E2E que reutilizan el TC.

Un átomo humilde como authenticateSuccessfully puntúa ROI base 1.5 — al borde de Deferred. Reutilizado en 5 flujos E2E: 1.5 × (1 + 0.2×5) = 1.5 × 2.0 = 3.0 → ahora un Candidate digno de automatizar. Los componentes reutilizables se ganan su lugar por composición, no por valor en solitario.

Por esto importan los test cases atómicos y componibles. Un paso de login que nadie automatiza de forma aislada se vuelve esencial cuando cinco flujos dependen de él. Piensa en los TC como componentes reutilizables.

No hay un cuarto bucket

Si >50% aterriza en Candidate/Manual, reaplica la Fase 0 con más rigor. La mayoría de los escenarios deberían ser Deferred.

Tres buckets, sin excepciones. En jira-native, Deferred ni siquiera crea un issue en el TMS — solo documentas la decisión en el reporte de priorización. En jira-xray el Test de sprint ya existe desde la Etapa 1 de sprint-testing, así que Deferred ahí significa que queda como artefacto de sprint sin promover, no borrado. Si demasiado está sobreviviendo, tu filtro es demasiado suave.

Mismo veredicto, distinta acción

La modalidad cambia el verbo, no el veredicto

El resultado de ROI (Candidate / Manual / Deferred) es idéntico en ambos mundos. Lo que cambia es lo que la Fase 3 hace con él.

jira-native — CREAR

  • Aún no existen Tests de sprint
  • La Fase 3 crea un work item Test solo para Candidate + Manual
  • Deferred = solo reporte, no se crea issue
  • El mismo pipeline Analizar→Priorizar→Documentar
vs

jira-xray — PROMOVER + ENRIQUECER

  • Los Test ya existen desde la Etapa 1 de sprint-testing
  • La Fase 3 selecciona + promueve los dignos de regression al Regression Test Plan + Test Set
  • y los enriquece (Gherkin rico, parametrización, casos límite)
  • Los Tests de sprint Deferred quedan sin promover, no borrados
Este es el matiz que más se pasa por alto. La aritmética del veredicto es compartida; el verbo es específico de la modalidad. En nativo, escribes los Tests desde cero para los sobrevivientes. En Xray los sobrevivientes ya nacieron como artefactos Test de sprint en la Etapa 1 — tu trabajo es elegir los dignos de regression, agregarlos al Regression Test Plan y al Test Set de la feature, etiquetarlos como regression-candidate, y especificarlos mucho más a fondo. Deferred nunca es un borrado en ninguno de los dos mundos.

Por bug — la regla que vale oro

Un bug digno de regression DEBE terminar con un Test

No todo bug se vuelve un TC de regression — pero si lo hace, debe terminar con un Test. Reutiliza el Test fallido existente si el bug vino de uno; si no, crea uno. Ambas modalidades.

❌ NO califica → Deferred

  • typo de una sola vez en un área estable
  • tratado como un test fallido — el fix ya fue verificado en sprint-testing
  • sin Test nuevo, regístralo como Deferred
→

✓ Digno de regression → debe tener un Test

  • vino de un Test fallido existente → REUTILÍZALO, enlaza + promueve, nunca dupliques
  • sin Test previo → CREA + diseña uno
  • enlaza vía tests / is tested by

Anula "el bug es el test case"

Esa frase de sprint-testing cubre solo el retest inmediato dentro del sprint — NO la regression futura. Esta regla decide si debe existir un Test persistente para que el bug nunca pueda regresar en silencio.

Regla de oro: donde existe un bug importante, un test debe atraparlo de nuevo. Un bug cerrado es prueba empírica fuerte de que el área regresa, así que la mayoría califica y se inclina a Candidate — pero no todos. El árbol de decisión: ¿candidato de regression? No → trátalo como un test fallido, Deferred, listo. Sí → ¿vino de un Test ejecutado existente? Reutiliza ese. ¿Sin Test previo? Crea y diseña uno. Nunca reabras un bug cerrado para adjuntar el TC — crea un TC nuevo y enlázalo.

PARTE 01 · FASE 3

Documentar
crear y enlazar en el TMS

Los escenarios que sobrevivieron se escriben en Jira/Xray como artefactos reales, enlazados y trazables — en un orden estricto.

La documentación es mecánica pero implacable con el orden. Equivócate en la secuencia y crearás huérfanos que solo una herramienta de reparación de trazabilidad puede arreglar.

No confundas los tres contenedores

Epic · Test Set · Test Plan — tres cosas distintas

Regression Epic

El paraguas del repositorio — uno por proyecto. Todo TC documentado vive bajo él. type=Epic, labels=test-repository.

paraguas

Test Set solo xray

Agrupación por feature, 1:1 con el Epic/feature. Contiene los Tests de regression promovidos de UNA feature. Equivalente nativo = Regression Epic + un label de feature/Epic.

feature

Test Plan

Alcance de ejecución / regression — el conjunto de Tests corridos para un release o ciclo de regression. El ATP en jira-xray es un issue Test Plan.

ejecución

Auto-crea el Test Set de forma perezosa (pregunta primero)

jira-xray: si el Test Set de la feature falta cuando una promoción lo necesita, la IA lo crea de forma perezosa en la Etapa 4 — pregunta al usuario primero, espejando la regla de preguntar-antes-de-crear del Regression Epic. Solo se agregan los Tests Candidate/Manual promovidos; los Tests de sprint Deferred no.

La gente fusiona estos rutinariamente. El Epic es todo el repositorio. Un Test Set agrupa los Tests de regression de una feature, uno-a-uno con el Epic de la feature. Un Test Plan es un alcance de ejecución. En jira-native no existe la entidad Test Set en absoluto — logras la misma agrupación con el Regression Epic más un label de feature/Epic en cada TC. La IA solo crea el Test Set de forma perezosa, cuando una promoción realmente lo necesita, y siempre pregunta antes de crear.

El orden no es opcional

ATP → ATR → enlazar → LUEGO cada TC

Orden de enlazado

(1) Crea ATP, enlaza a US. (2) Crea ATR, enlaza a US. (3) Actualiza ATP, enlaza a ATR (bidireccional). (4) Por cada TC: créalo, luego enlázalo a US + ATP + ATR + AC. (5) Por cada TC promovido — xray: agrégalo al Test Set de la feature + al Regression Test Plan + label regression-candidate; nativo: aplica el label de feature/Epic (sin entidad Test Set).

Para cuando se crea un TC, tanto ATP como ATR ya existen, así que los tres enlaces aterrizan en la creación. Crea los TC primero y obtienes huérfanos — y fix-traceability se vuelve la única salida. El modelo es bidireccional: US↔ATP↔ATR↔TC. Dado cualquiera, debes alcanzar los otros tres — ESO es trazabilidad.

Este es el orden que la gente rompe bajo presión de deadline. Plan y resultados primero, luego los tests. La cardinalidad también importa: US↔ATP y US↔ATR son 1:1; un ATR nunca se enlaza a dos ATP.

El artefacto — Gherkin de alta calidad

Variables, nunca datos hardcodeados

TC-GX-101.feature
@critical @regression @automation-candidate @GX-101
Scenario Outline: Validate authentication error when password is incorrect
  """
  Bugs covered: GX-202   Related Story: GX-101   ROI Score: 4.0
  """
  # === PRECONDITIONS ===
  Given a verified user exists with {email} in the database
  # === ACTION ===
  When the user submits login with {email} and {wrong_password}
  # === VALIDATIONS ===
  Then a 401 error is returned with format "<message>"
  # === EQUIVALENT PARTITIONS ===
  Examples: Wrong password | Empty password | Unknown email
La tabla de variables es obligatoria: {email} → SELECT email FROM users WHERE active=true LIMIT 1. Los tags siempre llevan prioridad + suite + flag de automatización + ID de Story. Nunca pegues un UUID o email literal — los datos de staging cambian a lo largo de los años. Los comentarios estructurados y la metadata del docstring hacen el TC autoexplicativo y automatizable.

Sin saltos

Draft no puede ir directo a Automated. Rework solo vía back_from_<state>. Deprecated es alcanzable desde cualquier estado. Manual puede reingresar a In Review si el ROI cambia.

El ciclo de vida es estrictamente secuencial. Recuerda dos campos independientes: Test Status (workflow — dónde está en su vida) versus Execution Status (la run — ¿pasó la última vez?). "Automated" + última run "FAIL" es una combinación normal y válida.

Termina el trabajo limpiamente

Labels, la regla de las dos llamadas de Xray, y la caché local

  • Label de alcance (requerido). regression casi siempre; smoke solo para rutas críticas (10-20% de la suite); más e2e/integration/functional.
  • Labels de estado. automation-candidate ↔ manual-only son mutuamente excluyentes; cambia a automated una vez mergeado.
  • Xray necesita DOS llamadas. una Create Test (registra) LUEGO una Update Issue para la Description — sáltate la segunda y el TC no tiene docs legibles.
  • Nunca escribas ADF JSON a mano. usa la ruta markdown→ADF; el ADF hecho a mano deriva y rompe los renderers.
  • Cachea localmente. un markdown por TC bajo .../stories/STORY-<KEY>/test-cases/{TC-ID}.md para un handoff limpio.
Estos detalles de cierre separan un repositorio limpio de uno desordenado. El patrón de dos llamadas de Xray es el fallo silencioso más común — un test registrado con cuerpo vacío.

PARTE 02

La Aceleración
conducirlo desde la terminal

Todo en la Parte 1 — el control de modalidad, las matemáticas de ROI, el orden de enlazado, las transiciones de workflow — el skill /test-documentation lo ejecuta desde un solo comando. Aquí está cómo, y cómo mantienes el control.

Ahora mapeamos el oficio manual sobre el skill. Mismas reglas, mismos controles — la IA solo los corre más rápido y nunca olvida el orden de enlazado.

Trigger + qué carga

Una frase, contexto completo cargado automáticamente

  • Trigger. /test-documentation, o frasing natural: "document tests", "ROI analysis", "which tests to automate", "Candidate vs Manual", "turn this bug into a regression test".
  • Chequeo de resume de Fase -1. lee .session/test-documentation/<scope>/progress.md; ofrece resume / restart / abort si una corrida previa se interrumpió.
  • Control de modalidad de Fase 0. auto-resuelve Xray vs Jira nativo (config → master-test-plan → listar issue types → preguntar).
  • Carga el contexto automáticamente. la carpeta sincronizada de la Story (story.md, acceptance-criteria.md, comments.md), los catálogos de Jira, y master-test-plan.md.
No pegas la metodología — el skill la lleva. Hasta revisa si una sesión de documentación previa se interrumpió y ofrece retomar exactamente donde se detuvo.

Modelo de orquestación

El hilo principal decide; los subagentes hacen el grueso de escrituras

Inline (orquestador)

La Fase 1 Analizar y la Fase 2 ROI quedan en el hilo principal; las decisiones y la planificación no se delegan.

criterio

Paralelo (subagentes)

Creación masiva de TC en Fase 3 cuando N > 10: fragmenta en bloques de ~5-10, tope de 10 subagentes (~10 escrituras/seg → evita 429s).

fan-out

Inline otra vez

El enlazado de trazabilidad y la matriz de cobertura final necesitan estado agregado, así que corren en el orquestador después de que los subagentes retornan.

agregar
La división es deliberada: el criterio vive en el centro de comando, las escrituras crudas de API se reparten. El tope de 10 subagentes existe porque tanto Jira como Xray throttlean alrededor de 10 escrituras por segundo.

Tres cosas para llevar a mañana

1 · Por defecto, Deferred

La mayoría de los escenarios validados NO deberían automatizarse. Corre los 3 filtros, luego ROI = (F×I×S)/(E×D). Mantén la suite pequeña y de alta señal.

2 · Una Pre+Acción = un TC

Nunca dividas aserciones en TC separados; nunca promuevas rasgos transversales a tests independientes; siempre valida el diseño contra el código real.

3 · Orden + trazabilidad sagrados

ATP→ATR→enlazar→TC; cada TC alcanza US/ATP/ATR/AC. El skill lo corre desde /test-documentation detrás de tu control de confirmación.

Si recuerdan solo tres cosas: Deferred es lo predeterminado, la identidad es precondición+acción, y el orden de trazabilidad es innegociable. La IA acelera el oficio — no reemplaza tu criterio.
