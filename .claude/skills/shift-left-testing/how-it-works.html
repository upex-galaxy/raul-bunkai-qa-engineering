PART 01

The Craft
Refining a Backlog by Hand

If the AI did not exist, what would a senior QA engineer do, step by step, to groom a batch of Stories before sprint planning?

Imagine no AI. Just you, a backlog, a Jira board, and a grooming meeting tomorrow. Here is the discipline.

position in the pipeline

Shift-Left vs. In-Sprint Testing

Shift-Left · Stage 0 (this activity)

  • Pre-sprint, batch of N Stories
  • Entry: Backlog / Shift-Left QA / Estimation
  • Audience: PO + BA + tester
  • NO execution — the feature does not exist
  • Output: refined ACs + risk map + ATP DRAFT

Sprint Testing · Stages 1-3

  • In-sprint, ticket-by-ticket
  • Entry: Ready For QA
  • Audience: Dev + tester
  • Smoke + UI/API/DB exploration
  • Output: ATP + ATR + bugs + evidence
Know which game you are playing. Shift-Left is forward-looking and code-free. If you find yourself running the app, you are in the wrong activity.

the shape of the work

One Pipeline, Three Phases — Always In Order

0

Session Init

Resolve the candidate list; confirm business context exists.

1

Selection

Triage which Stories deserve refinement, and at what depth.

2

Refinement

Per Story — surface gaps, refine ACs, draft outlines.

3

Handoff

Push to Jira, label, transition to Estimation, write the batch report.

Memorize this spine: Selection, Refinement, Handoff. Everything else hangs off these three. Never refine before you select; never hand off before you refine.

phase 0 — before you start

Three Things You Need Before Triage

  • A candidate list. Either explicit Story IDs (UPEX-100,101,102) or a backlog JQL query you build by hand.
  • Business context on hand. business-data-map, business-feature-map, business-api-map, master-test-plan. Without domain vocabulary your PO questions are vague and worthless.
  • A batch size of 1-12. The sweet spot. Past ~10-12 Stories attention collapses and quality degrades — split big groomings into multiple sessions.
The fatal beginner mistake is refining without business context — you produce generic questions any junior could ask. Anchor every refined AC in a real entity and a real flow.

phase 0 — backlog mode

The Candidate JQL

candidate-query.jql
project = UPEX
AND issueType = Story
AND status in (Backlog, "Shift-Left QA", Estimation, "Ready For Dev")
ORDER BY priority DESC, created DESC

# optional layers
AND sprint in openSprints()        # next-sprint candidates
AND "Epic Link" = EPIC-KEY          # groom one epic
AND updated >= -14d              # recent additions
When the user just says "groom the backlog," this is the query. Note the four entry statuses — those are the only states a Story can be in to qualify for Shift-Left.

phase 1 — the hard type filter

Stories Only — Reject Everything Else, Loudly

❌ Silently process or drop

  • Refine a Bug — reactive, no upstream ACs
  • Refine a Spike — no implementable ACs
  • Silently delete a mislabeled Tech-debt
→

✅ Loud reject with reason

  • UPEX-115 [Bug] — Shift-Left refines only forward-looking ACs
  • UPEX-127 [Spike] — refine the follow-up Story instead
  • UPEX-141 [Tech-debt] — no user-facing AC; convert to a Story first
The rejection is not noise — it IS value. Surfacing a mislabeled Bug tells the team their board is wrong. Never drop silently; always state the one-line reason.

phase 1 — the triage funnel

Five Gates Every Candidate Passes Through

1

Type filter

Reject non-Stories. Hard gate.

2

Label freshness

Already 'shift-left-reviewed' <30 days? Surface separately; ask skip/refresh.

3

Veto table

Money/auth/data-integrity force FULL; CSS/copy/tech-debt get SKIPPED.

4

Risk score

Only if no veto fired — sum factors into 0-3 / 4-7 / 8+.

5 · Data-feasibility scan

HIGH-risk only, doc-only — flags DATA-FEASIBILITY-RISK.

This is a funnel, in order. Type before label, label before veto, veto before score. The veto SHORT-CIRCUITS the score — that is the most important sequencing rule in selection.

phase 1 — veto beats score

The Veto Table — Force or Skip, Override the Math

VerdictTriggersWhy
FORCE FULLMoney, billing, refund, currencycalculation + audit trail
FORCE FULLAuth / authz / session / RBACsecurity perimeter
FORCE FULLData integrity on core entitiespersistent corruption risk
FORCE FULLExternal integrations (Stripe, Auth0)contract drift

Veto fires BEFORE you score. A refund feature is mandatory-refine even if its risk score is low. A CSS tweak is auto-skip even if it touched five files. Judgment overrides arithmetic.

phase 1 — the risk score

Memorize the heavy hitters: new feature and dynamic data are +3 each. AC-light is +2 — a thin Story scores high precisely because it needs you most. Comment-storm is +1 — noisy threads mean the team never agreed.

phase 1 — reading the score

Score Bands → Refinement Depth

ScoreLevelAction
0-3LOWSKIP — PO/Dev can write ACs without QA help
4-7MEDIUMStandard refinement (Phases 1-3 + outline-only Phase 4)

Score is advisory

The user may override a LOW skip into the set if they smell hidden complexity.

The band sets your DEPTH, not just in-or-out. HIGH does not mean 'do it harder randomly' — it means lean into Phase 2's ambiguity and edge-case hunt and widen the question set.

phase 1 — the gate

STOP — Present the Candidate Table and WAIT for OK

Hard human-in-the-loop gate

Selection ends with a ranked candidate table: Accepted (Story / Risk / Score / Veto override / Data-feasibility / Notes), Rejected non-Stories, Skipped LOW-risk, Already-Reviewed — plus a recommended refinement order and open questions.

You DO NOT start refining until the human confirms the set. The user can veto a Story, override a LOW skip, or change the order.

Selection is a proposal, not a decision. The table goes to the human, you wait, they OK or override. This gate is non-negotiable — it keeps the human steering the batch.

PART 01 · PHASE 2

Refinement
The Heart of Shift-Left

Now the real craft: per accepted Story, take a vague backlog item and turn it into a testable, gap-free, estimable Story.

Selection decided WHAT to refine. Phase 2 is HOW. This is where a QA engineer earns their seat at the grooming table.

phase 2 — the five sub-passes

The Refinement Playbook — Per Story

1

Critical Analysis

Business + technical context, complexity rating (LIGHT code reads only).

2

Story Quality Analysis

Ambiguities, gaps, edge cases, contradictions, testability.

3

Refined ACs

Given / When / Then with specific data.

4

Test Outlines

Coverage estimate + outline NAMES only.

5

Edge Cases

Names + criticality only.

This is the engine reused from sprint-testing's acceptance-test-planning — but in DRAFT mode. The whole skill deliberately reuses ~70% of that planning logic rather than reinventing it.

phase 2 — step 1

Critical Analysis — Context Without Reproduction

  • Business context. Primary/secondary personas, value proposition, KPIs influenced, position in the user journey.
  • Technical context. Frontend components/routes, backend endpoints (cited from business-api-map), DB tables, external services, integration points.
  • Complexity rating. Rate Business logic / Integration / Data validation / UI as Low/Med/High — this drives Step 2 depth and feeds the PO's estimate.
The hard rule here is LIGHT code reads. Read the entry files and one or two neighbors to confirm 'a senior dev would call this feasible.' You are NOT tracing call graphs or reproducing — that is in-sprint work.

phase 2 — step 2 (the heart)

Story Quality Analysis — Five Things You Hunt

Find the holes

  • Ambiguities — location in Story + the question + impact on testing + suggested clarification
  • Gaps — missing AC / business rule, why critical, what to add, risk if omitted
  • Edge cases not in the Story — scenario + best-guess behavior + criticality

Validate + reconcile

  • Contradictions — 'Description says X but AC3 says Y', surfaced explicitly
  • Testability — verdict Yes / Partial / No, listing every vague AC, missing error message, or un-isolatable dependency
This is THE step. PO can still act, so push harder than you would in-sprint. Every ambiguity here is a production defect you are deleting before it is born.

phase 2 — step 2 toolkit

The Question Catalog — Walk the Archetypes

ArchetypeA killer question it forces
Auth/SessionResponse on unauthenticated access — redirect, 401, 403, or 404? (403 vs 404 leaks existence)
Money/BillingCurrency precision + rounding rule? Off-by-one cent across millions of transactions
Search/ListEmpty state distinct from 'still loading'? Pagination skip/dupe on insert?
State MachineALL valid states + transitions? Behavior on an invalid transition?
IntegrationTimeout + retry policy? Idempotency-key strategy on retry?
Permissions/RBACOwner-vs-other — own records only or any in scope?

Plus 10 Universal questions (exact error text, success surface, idempotency, audit, rollback…).

A Story usually hits 2-3 archetypes. Walk each archetype's list; any question the Story does not already answer becomes a candidate gap. This catalog encodes failures testers usually only discover when execution fails.

phase 2 — step 2 classification

Triage Every Gap Into One of Three Buckets

1

CRITICAL

Blocks sprint planning. Goes into 'Critical Questions for PO'. e.g. an auth gate (A1) or refund cap (M2).

2

IMPORTANT

Blocks implementation, not planning. Goes into 'Technical Questions for Dev'. e.g. which Auth0 SDK version is pinned.

3

EDGE

Testable but not blocking. Goes into 'Edge Cases not in Story', tagged NEEDS PO/DEV CONFIRMATION.

Misclassifying signals bad judgment. Putting a 'what does the success toast say' question next to an auth-bypass question in the Critical list tells the PO you cannot prioritize. Bucket with care.

phase 2 — step 3 worked example

A Refined AC — Given / When / Then With Real Data

shift-left-refinement.md
#### Scenario 1.2: Should reject login when OTP is incorrect (Type: Negative, Priority: High)
- Given: user with an active OTP requested <5 min ago
- When:  user submits code "000000" (wrong)
- Then:
    UI:           inline error "Invalid code"
    API:          401, body { error: "invalid_otp" }
    DB:           no session row created
    System state: attempt counter +1

#### Scenario E1 (Type: Edge) — NEEDS PO/DEV CONFIRMATION
- behavior inferred — confirm before sprint planning
Refined ACs name exact inputs and assert across UI, API, AND DB. Anything you INFERRED — not literally in the original Story — carries the verbatim marker NEEDS PO/DEV CONFIRMATION so the PO sees it during planning. Tooling later greps for that exact string.

phase 2 — steps 4 & 5 the boundary

Outlines Yes, Test Data No — The DRAFT Discipline

❌ Over-engineering pre-code

  • Parametrization tables
  • Per-outline test-data JSON
  • Numbered test steps
  • Faker generation recipes
  • Creating any TMS test entity — TCs, ATP/ATR issues, Test Plans — in EITHER modality (the feature does not exist yet; Stage 4 creates them later)
→

✅ Shift-Left DRAFT scope

  • Coverage estimate table (Pos/Neg/Boundary/Integration counts → drives PO estimation)
  • Outline NAMES: 'Should reject negative refund amount on POST /refunds'
  • One-line precondition + one-line expected per outline
  • Edge-case names + criticality only
The discipline that separates Shift-Left from real test design: you produce outline TITLES and a coverage COUNT, never the data. The feature does not exist — generating Faker recipes now is wasted work that gets redone in-sprint.

test-design doctrine — the core principle

Refinement pushes past the happy-path contract

1 · Passing ≠ probing

A satisfiable AC confirms the business condition. It doesn't make the feature robust, safe, or error-resistant.

2 · Floor, not ceiling

ACs are the baseline for "done". Refinement's job is to surface what lives above that line.

3 · Criterion vs outline

A refined AC (Given/When/Then) is the business assertion; the outline (Should <behavior> <condition>) is its exploration.

4 · 1 : N by default

A non-trivial AC implies several outlines. Collapse to one only with a written "trivially atomic" reason — never the default.

5 · Risk lives outside

The heart of shift-left: surface the boundaries, states, and anomalies the Story is silent on — as PO questions AND as outlines.

A gap is an outline

A gap is not just a question for the PO — it's a missing outline. Map each archetype to the technique it implies.

Canon: agentic-qa-core/references/test-design-doctrine.md. The coverage estimate the PO uses must reflect the real 1:N, not a minimized count.

phase 2 — the deliverable

The Output File — shift-left-refinement.md

One working file per Story

.context/PBI/epics/EPIC-<KEY>/stories/STORY-<KEY>/shift-left-refinement.md

Canonical section order (do not reorder — maps 1:1 to the Jira block the PO reads): Critical Analysis · Story Quality Analysis · Refined ACs · Test Outlines (DRAFT) · Edge Cases (DRAFT) · Quality Assessment · Critical Questions for PO · Technical Questions for Dev · Suggested Improvements · Data-feasibility flags.

Empty sections STAY, marked 'None identified' — easier to verify completeness than to spot an absent section.

This is the historical record of pre-sprint analysis. Section order is load-bearing because it becomes the Jira description the PO grooms from. And present each Story's summary to the human, wait for OK, before moving to the next.

phase 2 — the professional's instinct

A Clean Story Is a Valid Outcome

❌ Both failures, not one

  • Padding: force a question or duplicate outline to look thorough
  • Under-deriving: treat a 1-line AC that hides a range or a state as a single scenario
  • Re-ask what the AC already answers
  • Put project-wide answers as PO questions when business-data-map already defines them
→

✅ Explode, then justify any collapse

  • Default 1:N — derive every outline the AC implies, then collapse only with a written reason
  • 4 ACs over a range or a state machine legitimately yield many more than 4 outlines
  • Empty question list when the Story is genuinely clear — say 'Story quality: good'
  • Cite business-data-map and skip the PO ask; the reader's bandwidth is the scarcest resource in grooming
This is the anti-pattern that separates seniors from juniors. Padding AND under-derivation both destroy trust: inflated counts train the team to skim, while a collapsed 1:N hides real risk. Shift-Left adds value by surfacing every REAL scenario the AC implies — neither more nor fewer.

phase 3 — handoff per Story

Six Jira Mutations to Hand Off a Story

  • 1 · Write refined ACs to the acceptance_criteria field (fallback: structured comment).
  • 2 · Append 'QA Refinements (Shift-Left Analysis)' to the description; populate the ATP DRAFT field with the full body.
  • 3 · Post ONE handoff comment notifying PO/Dev (mention handles only if in project.yaml).
  • 4 · Add BOTH labels — shift-left-reviewed + shift-left-{YYYY-MM-DD}.
  • 5 · Transition Backlog → Shift-Left QA → Estimation. 6 · Verify trace — field + comment are byte-for-byte mirrors.
Jira is canonical — there is no git commit. The local refinement file is the source for what you push, and the field/comment must mirror it byte-for-byte because fix-traceability checks this later.

phase 3 — the guardrail

NEVER Advance Past Estimation — and Always Both Labels

Stop at Estimation

From Backlog apply Analyze (→ Shift-Left QA) then Estimate (→ Estimation). PO + Dev lead own Estimate → Ready For Dev and sprint commitment — THEIR decision, not QA's. Already past Estimation? Log a warning, SKIP the transition; the refinement still lands.

Both labels, populated DRAFT

'shift-left-reviewed' is the marker /sprint-testing reads to short-circuit; 'shift-left-{date}' is the freshness stamp (still <30 days). Never transition to Estimation without a populated ATP DRAFT — the DRAFT is what makes the Story estimable.

Two non-negotiables. One: you never estimate or commit a Story — that is the PO's job, so you stop at Estimation. Two: both labels every time, or the whole short-circuit machinery downstream breaks.

phase 3 — the closing artifact

The Batch Report — One Session-Level Deliverable

What it aggregates

  • Session metadata (candidate/accepted/rejected/skipped counts)
  • Per-Story line: risk, gaps, critical PO Qs, tech Qs, outline count, status, trace
  • Aggregated Critical PO questions, DEDUPED and ranked by how many Stories each blocks
  • Risk distribution (HIGH/MEDIUM/LOW) + data-feasibility blockers

Where it goes

  • Recommended sprint-planning order, by risk + dependency
  • Posted as a comment on the parent epic IF all Stories share one
  • Otherwise delivered inline as the session-closing message
  • Pointer: when a Story hits Ready For QA, /sprint-testing short-circuits via the label
The batch report is the one thing the grooming meeting actually reads. Deduped, ranked PO questions are gold — 'this one question blocks three Stories' is exactly how a PO wants to triage.

PART 02

The Acceleration
One Command Runs the Whole Craft

Everything in Part 1 — selection triage, the refinement playbook, the question catalog, the six Jira mutations — driven by the AI from a single terminal invocation, with you in the loop at every gate.

You now know the craft. Here is how the skill makes the AI perform it end-to-end while keeping you in command.

how to invoke

The Trigger and What Auto-Loads

  • Trigger. /shift-left-testing, or natural language: 'shift-left these stories', 'groom the backlog', 'pre-sprint QA', or pasting Story IDs sitting in Backlog with 'refine these'.
  • Auto-loads context. business-data-map, business-feature-map, business-api-map, master-test-plan; the Story via bun run jira:sync-issues get <KEY> --include-comments.
  • Auto-loads tools. /acli always; /xray-cli only if Xray modality and you opt into a Test Plan.
  • Resolves modality. Xray-on-Jira (A) vs Jira-native (B) — and persists it to the session plan.
One phrase starts it. The skill self-loads the business context and the right Jira CLI so the AI is never flying blind on domain vocabulary or syntax.

orchestration model

Command Center + Specialist Subagents

Selection (Single)

Pulls candidates, applies veto + risk-score triage, returns the ranked table.

phase 1

Refinement (Sequential)

ONE per Story — runs the playbook, writes shift-left-refinement.md, returns a compact JSON summary. NEVER parallel; the human OKs each Story.

phase 2

Handoff + Batch-Report

Handoff does the six Jira mutations per Story; Batch-Report aggregates deduped questions and posts to the epic.

phase 3
The main thread is the command center; subagents are executors. Refinement is deliberately SEQUENTIAL — parallelism would burn your attention and break the per-Story approval rhythm the grooming cadence depends on.

artifacts + control

Where Everything Lands — and Who Stays in Charge

  • Per-Story refinement → .context/PBI/epics/EPIC-<KEY>/stories/STORY-<KEY>/shift-left-refinement.md (gitignored working file).
  • Jira → refined ACs in the field, ATP DRAFT, handoff comment, two labels, status at Estimation — the canonical source of truth.
  • Batch report → session folder + posted to the parent epic; then the session is archived.
  • You stay in control. Confirm the set, OK each Story summary, no transition past Estimation, no git commit, no test execution. The AI proposes; the human disposes.
The skill never runs the app, never queries a DB, never advances a Story into a sprint. It surfaces risk and hands off. Every consequential step is gated by your explicit approval.
