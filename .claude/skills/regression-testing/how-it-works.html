why this matters

20%

One misclassification flips the whole verdict

A test is only FLAKY above a 20% failure rate over the last 10 runs — below that line, a failure is a REGRESSION that BLOCKS the release. Call it "flaky" by reflex and you ship the bug.

The whole discipline hinges on not lazily tagging failures 'flaky'. The 20% threshold is the bright line between 'block the release' and 'don't'.

PART 01

The Craft
running a release gate by hand

If the AI did not exist, what would a careful QA engineer do, step by step, to turn a CI run into a release decision nobody can argue with?

Everything in this half is the real method the skill encodes — same thresholds, same trees, same templates.

the pipeline

Three phases, always in this order

1

Execute

Pick the right suite, trigger CI, monitor to completion, capture the run ID.

2

Analyze

Download artifacts, parse results, compute metrics, classify every failure, assess severity.

3

Report & Decide

Score the run, apply hard vetoes, emit GO/CAUTION/NO-GO, write the stakeholder report.

Stress the ordering rule: never skip analysis and jump to a report; never guess a classification without reading the failure logs. The order is non-negotiable.

phase 1 · choose

Pick the right suite before you trigger anything

SuiteWorkflowDurationUse when
regressionregression.yml20–60 minPre-release validation, nightly full run
smokesmoke.yml2–5 minPost-deploy health check, @critical only
sanitysanity.yml1–10 minOne feature / one file / one grep pattern

Default rule

Bare "run regression" → regression on the default env · "critical only" → smoke · a file or grep → sanity.

The wrong suite gives the wrong signal — a 3-minute smoke pass is NOT a release gate. Match suite to the question being asked.

phase 1 · execute

Trigger, then capture the run ID — in that order

1

Preflight

Confirm CI auth and the workflow exists (gh auth status, gh workflow list). Not authed → STOP, ask the user to log in.

2

Trigger

Dispatch the workflow with explicit inputs (environment, generate_allure=true).

3

Capture run ID

Wait 3–5s (the run isn't queryable instantly), then read the newest run's databaseId.

4

Persist

Write the RUN_ID down first; everything downstream keys off it.

The race condition is real — gh workflow run returns before the run registers, so you must sleep 3-5s. Losing the RUN_ID means you can't re-attach to a 60-minute run.

phase 1 · monitor

Watch to completion — don't analyze a half-finished run

Watch with an exit-status flag

Status moves queued → in_progress → completed. Only when completed do you get a conclusion: success / failure / cancelled / timed_out. On a long suite that times out, fall back to polling every 60–90s.

Rule

Do not start Phase 2 until status = completed. A cancelled run is not a failure — surface it and wait for direction, never silently re-trigger.

Half-read runs produce garbage verdicts. The conclusion field only exists once the run completes — that's your green light to analyze.

phase 2 · collect

Pull the evidence before it expires

1

Failed logs only

Pull only failed-step logs — the full log can exceed 50MB; --log-failed is the small, signal-rich slice.

2

Three artifacts

Merged Allure results, failure evidence (screenshots/traces/videos), and the Playwright HTML report into ./analysis/.

3

Previous run

Download the prior run's Allure into ./analysis/previous/ for trend.

Rule R4

Never skip the artifact download on red builds — evidence vanishes after the retention window.

On a red build the screenshots and traces are your only proof; once retention expires they're gone forever. Grab them first, analyze second.

phase 2 · parse & measure

Trust Allure first, then compute six numbers

  • Source-of-truth priority. Allure results JSON > Playwright report.json > raw logs; each result carries status, statusDetails, and labels (testId = ATC ID, suite, severity).
  • Pass Rate & Counts. Passed / Total × 100; plus Passed / Failed / Skipped / Broken by status.
  • Duration & Trend. max(stop) − min(start); trend = current pass rate − previous run's pass rate.
  • Retry caveat. A passed result with retriesCount > 0 is a hidden flake — count it even though Allure shows green.
Six numbers, one source-of-truth order. The retry-pass rule is the subtle one: green-on-retry still means unstable.

PART 01 · THE HEART

Classify Every Failure
the 5-way triage

This is where releases are won or lost. Every red test gets exactly ONE label, and the label decides whether it blocks the release.

This is the single most important slide group. Slow down here.

phase 2 · classify

Five categories, one label per failure

REGRESSION

Passed recently, now fails consistently after a code/data change.

HIGH · blocks release

FLAKY

Intermittent (>20% over last 10 runs) on an unchanged build.

MEDIUM · doesn't block

KNOWN ISSUE

Tracked by an existing ticket.

LOW · documented

ENVIRONMENT

Infra/network/external, not the app.

MEDIUM · re-run

NEW TEST

First-ever run, no history.

LOW · verify manually

The rule

A failure belongs to exactly one category — never improvise.

one label
Memorize these five. The whole release gate is built on getting each red test into the right bucket.

phase 2 · triage tree

Walk the tree top-down — stop at the first match

1

Known ticket?

→ KNOWN ISSUE.

2

Env error pattern?

ECONNREFUSED, ETIMEDOUT, net::ERR_, Navigation timeout, browserType.launch, 502/503 → ENVIRONMENT.

3

First-ever run?

→ NEW TEST.

4

Rate > 20% over last 10?

→ FLAKY.

5

Passed in last ≤5, now fails?

→ REGRESSION (release blocker).

Order matters — it's a precedence ladder, not a menu. The golden rule: NEVER mark a test REGRESSION without checking history first — that's the #1 misclassification.

phase 2 · precedence

When two rules match, precedence breaks the tie

Order: KNOWN ISSUE > ENVIRONMENT > NEW TEST > REGRESSION > FLAKY

If a ticket already tracks it, stop looking. If the error is clearly infrastructure, don't blame the code. A test with no history cannot be a regression yet. Only after ruling all of those out do you weigh REGRESSION vs FLAKY on the 20% line.

This ordering is deliberate. It pushes you to exhaust the cheap, non-blocking explanations before you reach for the expensive 'block the release' label.

phase 2 · don't scapegoat infra

ENVIRONMENT is not a scapegoat

❌ Lazy ENV tag

  • ECONNREFUSED → "infra glitch", move on.
  • Navigation timeout on one page → blame the network.
  • Tag everything network-shaped as environment so nothing blocks.
→

✅ Disciplined

  • ECONNREFUSED + many unrelated tests red on the same host → ENVIRONMENT.
  • ECONNREFUSED on ONE endpoint others hit fine → REGRESSION in disguise.
  • Nav timeout: one page after deploy → REGRESSION; across many → ENVIRONMENT.
The scope test is everything: is it one test or the whole host? One test failing where others succeed on the same endpoint is a bug wearing an infra costume.

phase 2 · pattern lookup

The error message usually tells you the answer

Error patternLikely classNote

status 500/502ENV or REGRESSIONisolated → REGRESSION, many → ENV

Anti-pattern: never classify on the test NAME alone. Read the actual error text — it is usually definitive.

phase 2 · flakiness math

Computing the >20% flaky rate honestly

1

Gather

Results over the last 10 runs of the same workflow + env (need at least 5).

2

Exclude

Drop runs against a different build (different commit SHA) — want "same build, different outcomes".

3

Compute

failure_rate = failed / total; retry-aware = (failed + retried_passes) / total.

4

Threshold

>0.20 → FLAKY; ≤0.20 with a pass in last 5 → REGRESSION; N<5 → "INSUFFICIENT HISTORY".

Flakiness needs at least 5 (ideally 10) runs of history. Below that you write 'insufficient history' and revisit next sprint — you never guess a flaky tag into existence.

phase 2 · hard cases

REGRESSION vs FLAKY — the judgment calls

Green streak, one fail

Code deployed between last pass and now → REGRESSION. No deploy, same commit → FLAKY candidate, monitor next run.

Passes on retry every time

FLAKY — user sees green but the test is unstable. Stabilize it.

One assertion flakes

In a multi-assertion test, the test is FLAKY — fix the one timing-sensitive assertion, don't mark the whole thing REGRESSION.

Fails on webkit only

Browser-specific API → REGRESSION (compat broken). Layout-sensitive → check per-browser history, may be FLAKY.

These four are where experience shows. The deciding question is almost always 'did a code change land between the last green and this red?'

phase 2 · severity

Severity is orthogonal — a flaky checkout test is still CRITICAL

SeverityCriteria
CRITICALCore journey (login, signup, checkout, payment, core search) or any @critical-tagged test
HIGHMajor feature (profile, dashboard, primary filters, account mgmt)
MEDIUMSecondary feature (sorting, preferences, non-primary filters, notifications)
LOWAdmin-only, edge case, internal tools

Inputs

Test tags (@critical/@smoke/@regression), suite name (Auth/Payment = CRITICAL), linked TMS ticket priority.

Classification answers 'why did it fail'; severity answers 'how much do we care'. They are independent axes — never collapse them.

PART 01 · THE VERDICT

Score the Run
apply the vetoes, decide

All the analysis funnels into one number on a 9-point scale — but vetoes can overrule a high score in an instant.

Now we turn evidence into a defensible decision.

phase 3 · scoring

The 9-point weighted score

Factor+3+10−1−2−3
Pass Rate≥95%90–95%——<90%—
Regressions01–2 Low—1+ Med—Any High/Crit
Critical testsAll pass————Any fail
Flaky tests—≤34–5>5——

Maximum score = 9

Regressions and critical-test failures carry the heaviest negative weight — that's intentional.

Four factors, each weighted by how much it should move a release decision. Regressions and critical-test failures carry the heaviest negative weight — that's intentional.

phase 3 · thresholds

Score maps to one of three verdicts

GO — score ≥ 7

Release approved; mark the candidate, schedule a post-deploy smoke.

ship it

CAUTION — score 4–6

Manual review required; document the accepted risks and proceed deliberately with the team lead.

human review

NO-GO — score < 4

Block the release; assign regression issues, schedule fix verification, plan a re-run.

block
The score is a guide, not gospel — CAUTION explicitly hands the decision to a human who documents what risk they're accepting.

phase 3 · hard vetoes

Three vetoes overrule any score

Never auto-GO if…

(1) any @critical test fails · (2) any REGRESSION with HIGH or CRITICAL severity exists · (3) pass rate < 90%. Hard vetoes regardless of how high the weighted score is.

R2 & R6

R2: the quality gate is binary — known REGRESSION count > 0 means you cannot emit GO. R6: never blend smoke + regression results into one pass-rate number; they have different SLOs.

This is the spine of the whole method. A run can score 8/9 and still be NO-GO because one critical test went red. The veto always wins.

phase 3 · report

Turn the decision into a stakeholder report

1

Executive summary

Verdict + score/9 + one-line rationale.

2

Metrics table

Pass Rate, Regressions, Critical fails, Flaky, Duration — each vs its threshold (ok/warn/fail).

3

Failure details

One section per category, Regressions first (test | ATC ID | last-passed | issue | cause).

4

Trend + links + recs

Last-5-runs sparkline; workflow/Allure/issue links; Immediate / Short-term / Long-term actions.

Save to

.context/reports/regression-{env}-{date}.md

The report is the deliverable a PM actually reads. Verdict first, evidence underneath, recommendations at the end. Same shape every time so people learn to scan it.

phase 3 · file & sync

When it's NO-GO, open issues and close the loop

  • One issue per regression. Title "[REGRESSION] {test} failing in {suite}", labels regression,bug,automated-tests, body carries Test ID, suite, run ID, env, error block, evidence links, last-passed run.
  • R7. Never mark a test KNOWN without a Jira ticket linking the suppression to a tracking issue.
  • R5. Never trigger a regression run without pinning the commit SHA; a different commit is a different baseline.
  • TMS sync (optional). Push per-ATC results into the Test Execution when a TMS is configured.
A NO-GO that doesn't create assigned, traceable issues is just a complaint. Each regression gets a ticket with full evidence so a dev can pick it up cold.

phase 1–3 · anti-patterns

The mistakes that sink a release gate

❌ Sinks the gate

  • Tag FLAKY without re-running in isolation.
  • Emit GO with one known REGRESSION "because the score was fine".
  • Skip the Allure download on a red build.
  • Mark a first-ever failure as REGRESSION.
  • Run without pinning the commit SHA.
→

✅ Trustworthy

  • Re-run in isolation before calling flaky (R1).
  • Treat any REGRESSION as a binary block (R2).
  • Download evidence first, before retention expires (R4).
  • First-run failure = NEW TEST, verify, then reclassify.
  • Pin --ref <commit-sha> so the baseline is fixed (R5).
Each left-hand habit is a real way teams ship bugs or block good releases. The right column is the discipline that makes the verdict trustworthy.

PART 02

The Acceleration
the skill runs the whole gate

Everything in Part 1 — suite selection, monitoring, the 5-way triage, scoring, the report — driven from one prompt in your terminal.

Now we show how the regression-testing skill executes the exact method we just learned, with the human still owning the verdict.

part 02 · invocation

One trigger phrase loads the whole pipeline

  • Trigger. "run regression", "GO/NO-GO decision", "analyze test results", "quality report", "release readiness", or /regression-testing.
  • Auto-loads context. Workflow YAMLs, master-test-plan (Epic key + SLOs), playwright.config.ts (retry/shard), the previous run's Allure baseline, kata-manifest.json, jira-required.yaml.
  • Won't trigger for. Writing new regression tests (that's test-automation) or manually retesting one bug fix (that's sprint-testing).
The skill description is tuned so it fires on release-gate language but stays out of the way for test-writing or single-ticket retest work.

part 02 · orchestration

Main thread is command center; subagents do the heavy lifting

1

Trigger (Single, inline)

One shell call, no dispatch.

2

Monitor (Background)

One subagent watches gh run watch while the main thread preps the report scaffold and loads the rubric.

3

Download (Parallel ×3)

Three subagents, one per artifact (allure / evidence / playwright), capped at 3.

4

Classify (Parallel, cap 10)

Failures sharded into chunks of ~10, one subagent per chunk, JSON merged in the main thread.

5

Compute + Verdict (Single, inline)

Release decisions live on the main thread only.

The pattern is deliberate: long waits and high-volume classification fan out to subagents; the decision itself never leaves the main thread, because that's where the human is.

part 02 · resume

20–60 min

The killer feature: re-attach, don't re-run

If the Monitor subagent dies mid-watch but RUN_ID was persisted to plan.md, Phase 0 re-attaches via "gh run view <RUN_ID>" instead of re-triggering CI — saving a full regression run of wall-clock. Session state lives at .session/regression-testing/<env>-<date>/.

This is why persisting RUN_ID immediately matters. A crashed watcher costs zero CI time if the run ID was written down — the skill just reconnects.

part 02 · human in the loop

The AI proposes; the human owns the release

Plan-first & error protocol

Phase 0 writes plan.md before any CI trigger and surfaces resume/restart/abort on an existing session. Any subagent failure → STOP, report full context, offer retry/skip/abort — never auto-fix.

Coded vetoes

The verdict and issue-filing stay on the main thread under your eye. The hard vetoes (@critical fail, HIGH/CRITICAL regression, pass-rate <90%) are coded in so the AI cannot rubber-stamp a GO it shouldn't.

The acceleration is real but bounded — the skill can't be talked into a GO past a veto, and it stops and asks rather than papering over a failure.

// takeaways

Three things to carry out of here

1 · Read before you label

Every failure gets one of five labels from the precedence tree — never call REGRESSION without checking history, or FLAKY without the >20% / 10-run math.

2 · Vetoes beat scores

One @critical fail, one HIGH/CRITICAL regression, or pass-rate <90% is an automatic NO-GO regardless of the 9-point score.

3 · The skill runs it, you own the verdict

One prompt drives execute→analyze→report with subagents and gh/allure, but plan-first and hard vetoes keep the human in control.

If they remember nothing else: classify honestly, respect the vetoes, and trust the method — the AI just runs it faster.
